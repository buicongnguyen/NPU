Analog CIM SDK and Compiler: From Model to Running Code

This page turns the Analog CIM research into an implementable software architecture. It defines the public SDK, compiler IR and passes, target and calibration artifacts, simulator and hardware backends, runtime and firmware contracts, framework integration, tests, and the smallest vertical slice that should work before the project grows. It is a reference design, not a disclosure of iHW, Mythic, IBM, or Tenstorrent internals.

Documentation review

What the previous pages explained, and what was still missing for implementation

The existing pages remain useful for physics, evidence, co-design, and reuse decisions. The changes below remove ambiguous ownership and add the contracts needed to write and test code.

PriorityReview findingWhy it blocks real codeResolution in this guide
P1Model calibration, silicon characterization, and per-device calibration were discussed under one name.They have different producers, update schedules, security requirements, and consumers.Three separate schemas and lifecycle owners are defined.
P1The executable and serial-number-specific device state were treated as one artifact.That prevents portable AOT compilation and creates a rebuild for every replacement device.A device-class executable is bound to a separate device-state bundle at load time.
P1No stable SDK API, handle ownership, queue, event, buffer, or error contract was specified.Compiler, runtime, simulator, applications, and tests cannot evolve independently.A versioned C ABI, C++ HAL, and Python workflow are provided.
P1Accuracy simulation and performance simulation were not separated strongly enough.AIHWKit and CrossSim can model numerical non-idealities, but they are not a product cycle/energy model.Functional, statistical, cycle/resource, and hardware backends have distinct acceptance criteria.
P2PyTorch, ONNX, StableHLO, ORT, and PJRT appeared as equivalent starting points.Supporting several frontends before one executable path works multiplies semantic and build risk.The MVP starts with ONNX; ORT and PJRT are staged integrations.
P2The compiler pipeline had no repository ownership or executable acceptance test.A list of passes does not show which library owns state or when a feature is complete.Modules, CLI commands, CI lanes, and vertical-slice exit tests are defined.
P2Firmware validation and recovery were described but not represented as a command protocol.Raw or weakly checked descriptors can damage state, hang queues, or use stale calibration.A bounded command set, state machine, validation rules, and recovery policy are specified.
P3Patent evidence and implementation design occupied the same long page.Readers could confuse a proposed software design with a vendor-disclosed implementation.The patent page now labels its software mapping as proposed, links here, and makes this page the authoritative build specification.

Architecture recommendation: build a simulator-first inference stack around one stable executable contract. Keep framework adapters above the compiler, physical device state below the artifact, and vendor-specific hardware behind a HAL. Do not begin by forking TTMetal, implementing PJRT, or supporting a complete model before one statically shaped MatMul compiles and runs end to end.

Proposed file extensions are illustrative. The important contract is ownership. A model build may depend on a target class and characterization version, but it should not embed mutable calibration from a particular serial number. The runtime binds an accepted .acal bundle and rejects stale or incompatible state.

Compiler implementation

Start with upstream IR, then introduce hardware meaning in three small dialects

MLIR's declarative ODS and dialect-conversion infrastructure provide generated operation classes, verifiers, parsing, printing, and legality-driven lowering. Use them for infrastructure; do not import Tenstorrent target semantics into an ACiM backend.

IR stageRecommended representationAllowed knowledgeVerifier examples
Imported graphUpstream func, tensor, arith, linalg, quant, or a small normalized frontend layerTensor semantics, shapes, constants, source names, QDQ scalesRanks known where required, contraction dimensions agree, quantized zero is representable
acim_planLogical MVM/Conv regions plus candidate precision, scale domain, fallback, and error budgetTarget capabilities and measured operation-model estimates, not physical addressesEvery op has a legal implementation; scale and accumulator types are complete
acim_physicalArrays, rows/columns, digit slices, sign paths, ADC grades, phases, buffers, and calibration classTarget geometry, characterization, fault-reserve policy, and static schedule resourcesNo overlap, current and range limits pass, all slices reconstruct, resources fit
acim_commandBounded commands, dependencies, events, DMA, program/verify, execute, telemetry, and recovery pointsFirmware ABI and driver-visible memory handlesNo raw pointers, all buffer ranges valid, dependency graph acyclic, firmware supports opcode/version
Starter ODS shapeLogical MVM is pure; programming and command operations are not
def ACIMPlan_MvmOp : ACIMPlan_Op<"mvm", [Pure]> {
  let summary = "Logical matrix-vector operation eligible for ACiM planning";
  let arguments = (ins
    AnyRankedTensor:$activation,
    AnyRankedTensor:$weight,
    ACIM_PrecisionAttr:$precision,
    ACIM_ErrorBudgetAttr:$error_budget,
    ACIM_FallbackAttr:$fallback);
  let results = (outs AnyRankedTensor:$result);
  let hasVerifier = 1;
}

def ACIMCmd_ProgramArrayOp : ACIMCmd_Op<"program_array"> {
  let summary = "Request a bounded firmware-owned program and verify sequence";
  let arguments = (ins
    ACIM_ArrayRefType:$array,
    ACIM_ProgramImageType:$image,
    ACIM_CalibrationClassAttr:$calibration_class,
    ACIM_EventType:$dependency);
  let results = (outs ACIM_EventType:$completion);
  let hasVerifier = 1;
}

Implement passes in dependency order and save every boundary

OrderPass/libraryImplementation responsibilityMinimum test
1onnx-check-importRun ONNX checking and shape inference; import only MatMul/Gemm first; preserve source location and constants.Valid model imports; invalid rank/K dimension returns a stable diagnostic.
2canonicalize-contractsNormalize transpose, bias, broadcast, and contraction forms into one MVM semantic contract.Equivalent ONNX forms produce identical normalized IR.
3attach-quant-contractConsume QDQ or .qcal; materialize scale, zero point, clipping, rounding, and accumulator type.Integer software reference exactly matches expected output and saturation.
4analyze-capabilityGenerate reason-coded analog, digital, and host candidates from target capabilities.Unsupported shapes are rejected or receive an explicit fallback candidate.
5partition-and-fuseChoose regions using boundary conversion and transfer cost, not operation support alone.A forced analog/digital override produces a report and valid graph.
6decompose-weightsGenerate radix/sign slices and reconstruction coefficients; preserve logical-to-physical debug mapping.Reconstructed logical weights exactly equal the quantized reference.
7place-arraysTile matrices, reserve spares, constrain current, pack slices, and emit calibration-class requirements.No overlap; capacity and current constraints are checked by an independent verifier.
8plan-adc-phasesSelect ranges, converter sharing, phase count, repeat policy, and saturation/fallback guard.Every candidate reports predicted error, phases, saturation margin, and confidence.
9plan-memory-scheduleAllocate static arenas and overlap DMA, input encoding, MVM, ADC, reconstruction, and digital work.Lifetime intervals do not overlap illegally; cycle model accepts the schedule.
10lower-command-abiEmit versioned bounded commands and events; insert firmware validation and recovery checkpoints.Command schema round-trips and malformed descriptors fail before execution.
11serialize-acxPackage target requirements, programs, constants, plans, commands, reports, and debug maps.Artifact is deterministic, inspectable, schema-conformant, and rejects wrong target major versions.
12differential-verifyRun exact reference and selected backends; compare checkpoints and release thresholds.First divergent op, expected tolerance, seed, profile versions, and tensors are retained.

Candidate selection needs measured coefficients and uncertainty

Candidate record and constrained objectivePersist this data in compiler reports
candidate = {
  implementation: analog | digital | host,
  layout: {tiles, groups, buffers, sharding},
  numerics: {weight_digits, activation_bits, adc_grade, accumulator},
  physical: {arrays, phases, spares, calibration_class},
  estimates: {latency_ns, energy_pj, error_distribution, confidence},
  boundaries: {copies, conversions, synchronization},
  fallback: {trigger, implementation, recovery_cost}
}

minimize latency_weight * latency_p99
       + energy_weight  * energy
       + risk_weight    * accuracy_risk
       + boundary_weight * transfer_and_conversion
       + wear_weight    * programming_and_refresh

subject to error_budget, SRAM, current, ADC range, capacity,
           firmware ABI, calibration class, and service policy.

Do not use an accuracy simulator as a latency oracle. CrossSim explicitly states that it does not model accelerator energy, area, or speed. Use AIHWKit or CrossSim adapters for numerical studies, then maintain a separately correlated cycle/resource model for queues, SRAM, NoC, converters, firmware, and digital work.

API ruleRequired behaviorTest
OwnershipEvery successful create/open/load/allocate returns one owned reference. Parents outlive children; queues are drained or canceled before release; release is called exactly once.Sanitizer and fault-injection tests cover success, partial construction, outstanding events, and every failure path.
Thread safetyContext/device queries are concurrent; a queue is ordered; independent queues may overlap; session concurrency is reported by capabilities, never assumed.Multi-threaded submit and teardown with deterministic per-queue order and allowed cross-queue overlap.
AsynchronySubmit returns an event without waiting; buffer lifetime extends through event completion.Two queues overlap transfers and execution in the cycle and hardware backends.
ErrorsStatus codes are stable categories. Synchronous details are thread-local to the context; event results carry structured subsystem, device code, trace ID, and bounded message text.Firmware fault, timeout, stale calibration, cancellation, and incompatibility map consistently across backends.
VersioningThe caller initializes struct_size and api_version; the library copies input strings and ignores compatible unknown trailing fields.Old header/new library and new header/old library compatibility matrix plus malformed-size tests.
Tensor bindingExecutable metadata fixes input/output index, name, type, shape, and byte size. Every binding validates index, offset, size, and memory compatibility before enqueue.Wrong type/shape/order, undersized buffer, overlap, and stale-handle tests fail before device work begins.
Device stateThe trusted .acal bundle is signature-, family-, schema-, age-, and temperature-checked before executable load and programming.Wrong serial/family, expired calibration, revoked signature, and remap-capacity failures are reason-coded.
Memoryv0 exposes allocated host-visible, pinned, and device-local buffers with blocking read/write convenience calls. Zero-copy import is added only with explicit ownership, coherence, and synchronization rules.Alignment, bounds, DMA direction, cache maintenance, and IOMMU isolation tests.
CancellationCancellation is best-effort and reports whether execution stopped, completed, or required reset.Cancel before dispatch, during DMA, during MVM, and after completion.

Python should expose the same lifecycle without hiding events or reports

Python compile and runOne obvious path from ONNX to a simulator or device
from acim_sdk import Compiler, CompileOptions, Context, TargetProfile

target = TargetProfile.load("targets/rev_b.atarget", "targets/rev_b.achar")
compiler = Compiler(target)

artifact, report = compiler.compile_onnx(
    "models/linear.onnx",
    model_calibration="build/linear.qcal",
    options=CompileOptions(
        accuracy_budget=0.005,
        latency_target_us=100,
        allow_digital_fallback=True,
        save_intermediate_ir="build/ir",
    ),
)
artifact.save("build/linear.acx")
report.save("build/linear.compile.json")

with Context(backend="functional") as context:
    device = context.open_device(0)
    session = device.load("build/linear.acx")
    output = session.run({"input": input_tensor})

with Context(backend="device") as context:
    device = context.open_device(0)
    session = device.load("build/linear.acx")
    event = session.submit({"input": input_tensor})
    output = event.result(timeout=1.0)
    event.trace().save("build/linear.atrace")

Framework integration

Add framework plugins only after the standalone contract works

StageIntegrationWhat to implementWhy this order
MVPStandalone ONNX CLI and Python compilerChecker, shape inference, MatMul/Gemm import, static shapes, compile, inspect, runSmallest dependency surface; deterministic artifacts; easy golden testing
Stage 2ONNX Runtime plugin EPGetCapability, Compile, allocator/stream if needed, compatibility info, session optionsProvides heterogeneous CPU fallback and a production application API without owning the whole framework

Since ONNX Runtime 1.23, its official guidance is to implement a new provider as a plugin EP. The separate plugin development contract asks GetCapability() which graph regions are supported and Compile() to produce executable compute functions, while another provider can handle fallback. That is a good second integration after the standalone compiler is stable.

Execution Provider responsibilitiesKeep the plugin thin
GetCapability(graph, target):
  validate shapes and dtypes
  ask libacim_compiler for legal candidate regions
  subtract regions whose boundary cost or error budget fails
  return reason-coded supported subgraphs

Compile(subgraphs):
  compile each accepted subgraph to .acx bytes
  retain compatibility metadata and cache key
  create compute callback that calls the public acim runtime API

Compute(inputs, outputs):
  bind ORT buffers or copy through declared memory rules
  submit asynchronously where the EP contract permits
  translate acim status/faults to ORT status
  never implement quantization, mapping, or driver logic in the EP
