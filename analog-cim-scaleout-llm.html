Analog CIM Scale-Out, Fault Tolerance, and LLM Inference

A concrete software architecture for turning imperfect analog arrays into a logical accelerator mesh. It adapts selected Tenstorrent ideas at the descriptor, virtual-device, simulator, collective, and observability boundaries while replacing Tensix execution with calibration-aware Analog CIM placement and a hybrid digital backend.

Executive answer

Can a Mythic- or iHW-style Analog CIM system run an LLM?

Architecture conclusion Yes, as a hybrid inference system No public evidence yet establishes a legacy M1076 or an iHW product running a production decoder LLM end to end.

Put static dense projections and feed-forward weights on Analog CIM arrays. Keep dynamic attention, KV-cache writes, normalization, nonlinear functions, routing, sampling, and precision recovery on digital engines and SRAM/DRAM. Scale the model with explicit tensor, pipeline, and expert parallelism. Judge success in tokens/second, time-to-first-token, inter-token latency, accuracy, energy/token, capacity, and recovery time, not array TOPS alone.

Measured silicon

Transformer feasibility is real but bounded

A 2025 IBM-led study mapped 7.1 million unique ALBERT weights to a fabricated 14 nm PCM chip and reported average accuracy 1.8 percentage points below FP. Attention, activations, pooler, and classifier remained in software, so this is strong evidence for analog linear layers, not an end-to-end decoder LLM.

Read the Nature Communications paper
Circuit and model simulation

Dynamic attention needs different memory

A 2025 gain-cell proposal stores K/V values in write-friendly charge memory and computes both attention dot products in the analog domain. Results use 64 × 64 SPICE simulations and PyTorch model evaluation, not fabricated full-system silicon. It should be a separate target capability, not assumed for flash arrays.

Physical execution

What actually happens during one Analog CIM matrix-vector operation

Mythic publicly identifies a flash array and analog circuits, but not every internal encoding, converter, or accumulation detail. The flow below is a generic differential or bit-sliced ACiM contract whose exact choices must be supplied by the target descriptor.

1. ProgramEncode fixed weightsMap quantized weight slices into conductance states, verify writes, and record achieved values.Wq -> G+, G-
2. DriveEncode activationsDACs, pulse widths, or switched levels convert an input slice into row stimuli.xq -> V or t
3. MultiplyCell currentEach enabled cell contributes current according to its programmed state and applied input.Iij ≈ Gij * Vi
4. AccumulateColumn integrationKirchhoff summation and integration form one partial dot product per active column.Ij = ∑i Iij
5. ConvertRead analog resultADC or charge-to-pulse circuitry digitizes the integrated result using a calibrated range.qj = ADC(Ij)
6. RebuildDigital correctionCombine signs, slices, arrays, and chips; apply scale, bias, saturation, and optional residual correction.y = requant(sum(q))
y_hat = D(ADC(sum_i A(G_ij, x_i, theta))))

The software-visible result is not only W × x. A includes device and circuit behavior, theta identifies calibration and environment state, and D is digital reconstruction. A compiler must preserve all three in the executable and debug map.

Defect hierarchy

A bad cell, a bad ADC, and a bad tile are different problems

Each level needs a distinct health record and recovery action. Collapsing all defects into one bitmap makes placement unstable, wastes healthy capacity, and prevents useful diagnostics.

Important evidence correction: Mythic's public patent supports virtual addressing and harvesting of defective compute tiles, including removal of a minimal set of tile rows or columns. It does not by itself prove spare flash-cell rows, spare bitlines, or spare ADC channels inside Mythic's ACE array. The row/ADC example below comes from a separate EnCharge AI patent and is used only as an alternative design pattern.

Mythic patent mechanism

Virtualize the tile grid

Test tiles, mark low-performing or defective tiles, and create a physical-to-virtual coordinate map that maximizes usable compute. A null-address tile can be skipped as a compute destination while packet routing continues through it. Applications target a stable logical grid.

Buildable contracts

The compiler targets capabilities; the runtime binds a particular device

This is the key separation borrowed from virtual device and harvesting designs. A portable executable should express logical resource requirements and legal variants. Serial-number-specific defects remain in the binding record.

Stable C++ runtime boundary

class AcimDevice {
 public:
  virtual CapabilityView capabilities() const = 0;
  virtual HealthView health() const = 0;
  virtual Binding bind(const Executable&) = 0;
  virtual Event program(const Binding&) = 0;
  virtual Event submit(const CommandBatch&) = 0;
  virtual RecoveryResult recover(const Fault&) = 0;
};

class AcimMesh {
 public:
  virtual MeshTensor distribute(const Tensor&, ShardSpec) = 0;
  virtual Event collective(Collective, MeshTensor&) = 0;
  virtual Checkpoint checkpoint() = 0;
  virtual void reform(const TopologyView&) = 0;
};

Security and correctness: firmware validates descriptor bounds, executable signatures, ABI versions, programmed-weight hashes, calibration validity, DMA ranges, and route epochs. Host metadata is never trusted merely because it came from the compiler.

Fault-aware compiler

Placement is a constrained optimization, not a rectangular tiling exercise

Compiler and runtime placement algorithmCompile against capability classes; bind against one health epoch.
for op in topological_order(graph):
  variants = legal_lowerings(op, target, accuracy_policy)
  for variant in variants:
    shards = tile_tensor(variant, logical_mesh)
    candidates = resources_matching(shards, capability_class)
    score = (
        estimated_latency
      + lambda_energy * estimated_energy
      + lambda_error  * predicted_numerical_error
      + lambda_comm   * collective_bytes
      + lambda_wear   * programming_and_aging_cost
      + lambda_risk   * spare_consumption
    )
  choose pareto_feasible_variant_or_fallback()

at load time:
  verify health_epoch and calibration envelope
  match logical shards to physical arrays
  apply row/column/tile remaps
  program and verify weights
  publish signed binding record

on fault:
  retry -> recalibrate -> local remap -> tile rebind
        -> mesh reform/checkpoint restore -> digital fallback
Accuracy cost

Predict layer sensitivity

Use quantization sweeps, measured nonideality models, activation statistics, calibration state, and selective digital residuals. Sensitive layers can request wider ADC plans or digital execution.

Capacity cost

Reserve without wasting the fleet

Compile variants for nominal and degraded capacity classes. Keep spare rows, columns, arrays, and tiles out of normal placement unless policy explicitly spends them.

Communication cost

Place around collectives

Estimate bytes and hops for activation shards and partial sums. Local weights remove weight traffic, not all inter-chip traffic.

Research input: the 2026 Dyn-A-Map paper ranks vulnerable subarray regions and relocates important weights to healthier regions without retraining. Its evidence is pre-silicon SPICE and architecture simulation, so use it as a mapping heuristic to validate on real devices, not as a proven field-repair result. Read Dyn-A-Map.

Prefill

High parallelism favors array utilization

Many prompt tokens turn projection and MLP work into matrix-matrix execution. Tile tokens and output channels together, overlap digital attention with analog projections, and pipeline collectives. Time-to-first-token remains the product metric.

Decode

Weight locality helps, but dynamic state dominates

Each step is sequential and often matrix-vector at low batch. Converter latency, digital companion balance, KV-cache access, inter-chip activation traffic, and synchronization can dominate. Measure inter-token latency and energy/token.

Software pattern worth reusing: vLLM's PagedAttention shows why the KV cache needs a virtualized, dynamically allocated memory layer. Reuse the paging and serving concepts; replace GPU-specific kernels and memory APIs with the ACiM hybrid runtime. Read the PagedAttention paper.

Four-bit LLMs

Four logical bits do not automatically mean one cheap analog operation

Representation

Separate model bits from physical cost

  • Logical weight formatW4 may be symmetric/asymmetric and per-tensor, per-channel, or group-scaled.
  • Cell representationOne logical weight may use one multi-level cell, a differential pair, several devices, or bit slices.
  • Activation executionA8/A16 may require serial input slices and multiple integration cycles.
  • Output reconstructionADC width, partial-sum count, scale precision, and digital accumulation determine effective accuracy.
Accuracy policy

Use selective precision, not one global fidelity

  • Quantize with measured rangesCalibrate group scales and clipping using representative prompts and target nonideality.
  • Protect sensitive pathsKeep norms, accumulators, logits, outlier channels, and selected layers in higher-precision digital execution.
  • Train or adapt for hardwareUse noise-aware fine-tuning or a small digital LoRA/residual adapter when model access permits.
  • Promote on evidenceChoose more slices, wider conversion, repeat-and-average, or fallback only when accuracy gain exceeds latency/energy cost.

The M1076 brief reports capacity in "weight parameters," not packed bytes. A 4-bit software checkpoint cannot be assumed to double M1076 weight capacity. Physical packing depends on unpublished cell encoding, differential representation, redundancy, slicing, and metadata. The compiler must query a target-specific physical_slots_per_logical_weight function.

A recent transformer adaptation study places fixed base-model weights on analog tiles and small LoRA corrections on digital cores; it reports simulation through LLaMA 3.1 8B, not full-system LLM silicon. This is a useful software architecture for preserving programmable adaptation without repeatedly reprogramming NVM arrays. Read the AHWA-LoRA study.
