These terms identify different software layers, transport mechanisms, or numerical operations. Short reminders appear again beside the sections that use them so an acronym does not silently change meaning.

Do not confuse

Similar words describe different contracts

API vs ABI vs UAPIAn API is what source code calls. An ABI is the binary representation and behavior. A UAPI is specifically the kernel/userspace API and ABI surface.
Runtime vs UMD vs KMD vs firmwareThe runtime schedules model execution; the UMD translates userspace objects; the KMD enforces host protection; firmware controls device engines.
BAR/MMIO vs DMABAR-backed MMIO carries small control/register accesses. DMA moves bulk buffers between host memory and the device.
SQ/CQ vs doorbell/MSI-XSQ and CQ hold durable command/completion records. A doorbell and MSI-X only notify the other side that records are ready.
IOVA vs CPU pointer vs physical addressA CPU pointer belongs to a process, a physical address names RAM, and an IOVA is the device-visible address produced by the DMA/IOMMU mapping.
Quantization error vs transport errorADC, low precision, and calibration affect numerical accuracy. PCIe AER, DMA, ECC, and queue faults affect reliable delivery and execution.

Driver decision

Does an Analog CIM board need a user-mode driver (UMD) and kernel-mode driver (KMD)?

Terms here UMD = User-Mode Driver KMD = Kernel-Mode Driver HAL = Hardware Abstraction Layer UAPI = kernel/userspace contract
Architecture conclusion Production PCIe: yes A bus-mastering accelerator should not trust a process with raw physical addresses or unrestricted register access.

Keep policy, graph execution, artifact parsing, and batching in a UMD/runtime. Keep isolation, DMA mappings, interrupt ownership, reset, power, device lifetime, firmware loading, and the stable Linux UAPI in a KMD. Firmware must validate the device-side command contract again. A simulator or tightly controlled lab prototype can use a smaller path while the hardware protocol is still moving.

Software-only No KMD Use the same HAL against functional, statistical, and cycle backends. Mock BARs, queues, interrupts, reset, and faults. Best for: compiler and runtime CI
Trusted FPGA lab VFIO with an IOMMU Userspace owns BARs, DMA mappings, and interrupts through VFIO. Verify the complete IOMMU group, FLR or reset, and locked-memory limits. Best for: protocol exploration
Narrow prototype UIO only for simple MMIO UIO maps device memory and forwards interrupts, but has no IOMMU protection. Do not make it the product path for unrestricted bus-master DMA. Best for: mailbox or Programmed I/O (PIO) loopback
Product path Dedicated KMD + UMD/HAL Context-bound buffers, validated queues, MSI-X, reset, AER, firmware, telemetry, permissions, and backward-compatible UAPI. Best for: multi-process deployment

Subsystem choice: evaluate Linux drivers/accel before inventing a private device framework. The accelerator subsystem exposes /dev/accel/accel* and reuses DRM infrastructure. A small out-of-tree character driver is acceptable for early board enablement, but its UAPI should still be designed as if it must survive old userspace and new kernels.

Compile and deploy flow

Do not make MLIR responsible for PCIe

MLIR selects and encodes legal hardware work. The runtime resolves physical buffers and live device state. The KMD protects host resources. Firmware owns device scheduling. Keeping these decisions separate permits the same artifact to run in simulation and on a board.

First model dataflow

A YOLO demo should prove the whole hybrid path, not only one analog convolution

This placement is a proposed starting point, not a claim about an iHW or Mythic product. The exact YOLO version, supported operators, array geometry, digital engines, precision, and SRAM capacity must decide the final partition. Keep host fallback available until each device segment passes differential tests.

Command and queue ABI

Descriptors carry handles and bounded references, not trust

A KMD resolves process-owned buffer handles into IOVAs. Firmware accepts only ranges registered in the context table. Keep fixed-width fields, natural alignment, explicit reserved fields, and static size checks. Add a new ioctl or opcode for new semantics instead of silently changing an old one.

Terms here SQ = Submission Queue CQ = Completion Queue ABI = binary command contract IOVA = mapped DMA address Doorbell = MMIO queue notification
Proposed firmware queue entriesIllustrative v0 shared header; all fields little-endian
/* include/hw/acim_queue_v0.h - generated from the protocol schema */
struct acim_sq_entry_v0 {
  __le16 opcode;          /* EXECUTE, PROGRAM, VERIFY, COPY, TRACE, ABORT */
  __le16 flags;           /* interrupt, priority class, retry permission */
  __le32 command_bytes;   /* bounded command-buffer length */
  __le64 context_id;      /* firmware-created context, never a pointer */
  __le64 command_iova;    /* IOVA already registered for this context */
  __le64 user_cookie;     /* opaque value returned in completion */
  __le64 deadline_ns;     /* relative or zero; clock semantics are specified */
  __le32 generation;      /* rejects work submitted before reset */
  __le32 crc32;           /* optional transport corruption check */
};

struct acim_cq_entry_v0 {
  __le64 user_cookie;
  __le32 status;          /* OK, INVALID, FAULT, TIMEOUT, RESET, RETRY */
  __le32 fault_code;
  __le64 device_cycles;
  __le64 completed_bytes;
  __le32 generation;
  __le32 reserved0;
};

Kernel-mode driver (KMD) skeleton

Probe in dependency order; remove in reverse after quiescing DMA

The skeleton shows order and failure ownership, not drop-in production code. Select current managed PCI APIs for the supported kernel range, implement every unwind path, and test probe failure after each resource acquisition.

Terms here KMD = Kernel-Mode Driver IRQ = Interrupt Request DMA = Direct Memory Access Probe = Linux device initialization callback
Proposed Linux PCI probe pathEnable, claim, map, configure DMA, IRQ, firmware, then expose userspace
static int acim_probe(struct pci_dev *pdev,
                      const struct pci_device_id *id)
{
  struct acim_dev *adev;
  const struct firmware *fw = NULL;
  int ret, i;

  ret = pcim_enable_device(pdev);
  if (ret)
    return ret;

  ret = pcim_iomap_regions(pdev, BIT(0), "acim");
  if (ret)
    return ret;

  adev = devm_kzalloc(&pdev->dev, sizeof(*adev), GFP_KERNEL);
  if (!adev)
    return -ENOMEM;

  adev->bar0 = pcim_iomap_table(pdev)[0];
  pci_set_drvdata(pdev, adev);

  ret = acim_read_and_validate_caps(adev);  /* ROM-visible; no DMA yet */
  if (ret)
    return ret;

  if (adev->caps.dma_bits != 32 && adev->caps.dma_bits != 64)
    return -EPROTO;

  ret = dma_set_mask_and_coherent(&pdev->dev,
                                  DMA_BIT_MASK(adev->caps.dma_bits));
  if (ret)
    return dev_err_probe(&pdev->dev, ret, "unsupported DMA mask\n");

  pci_set_master(pdev);                 /* device may now initiate DMA */

  ret = acim_alloc_admin_rings(adev);       /* dma_alloc_coherent */
  if (ret)
    goto err_clear_master;

  ret = pci_alloc_irq_vectors(pdev, 1, adev->caps.max_vectors,
                              PCI_IRQ_MSI | PCI_IRQ_MSIX);
  if (ret < 0)
    goto err_free_rings;
  adev->nr_vectors = ret;

  for (i = 0; i < adev->nr_vectors; ++i) {
    ret = request_threaded_irq(pci_irq_vector(pdev, i),
                               acim_irq, acim_irq_thread, 0,
                               "acim", adev);
    if (ret)
      goto err_free_irqs;
  }

  ret = request_firmware(&fw, adev->caps.firmware_name, &pdev->dev);
  if (ret)
    goto err_free_irqs;

  ret = acim_boot_firmware(adev, fw);       /* reset, load, hash, start, ABI */
  release_firmware(fw);
  if (ret)
    goto err_free_irqs;

  ret = acim_register_accel_or_chardev(adev); /* userspace opens only now */
  if (ret)
    goto err_quiesce;

  return 0;

err_quiesce:
  acim_quiesce_and_reset(adev);
err_free_irqs:
  while (i--)
    free_irq(pci_irq_vector(pdev, i), adev);
err_free_vectors:
  pci_free_irq_vectors(pdev);
err_free_rings:
  acim_free_admin_rings(adev);
err_clear_master:
  pci_clear_master(pdev);
  return ret;
}
1. BlockStop new opens/submitsMark device removing or lost; wake waiters with a terminal error.
2. QuiesceMask and drainDisable queues, stop engines and DMA, synchronize IRQ handlers.
3. ReclaimFinish ownershipSignal/cancel fences, unmap streaming DMA, detach dma-buf and unpin pages.
4. ReleaseIRQ and ringsFree vectors, coherent control memory, firmware state and device node.
5. DisableClear bus masterConfirm no bus transaction remains before managed BAR/device release.

Important: do not enable processing engines or expose a device node immediately after pci_set_master(). First validate the register ABI, allocate protected control rings, install interrupts, boot compatible firmware, and complete a health handshake. A failed firmware boot is a detected device that is not ready for userspace.

DMA and interrupt dataflow

Map only for the device lifetime of the transfer, then complete ownership

Use coherent DMA for small long-lived queue/control structures and streaming mappings or managed buffer objects for payloads. The DMA address returned by the kernel is a device bus address, not a CPU physical address.

Terms here DMA = Direct Memory Access IOMMU = address translation and isolation IOVA = device-visible DMA address MSI-X = completion/fault notification
1. Create/importBuffer objectKMD allocates or imports pages and records owner, size, usage and direction.
2. MapIOMMU / DMA APICreate IOVA or scatter/gather mapping; check every mapping result.
3. PublishValidated descriptorWrite command and ring fields, transfer ownership, then ring doorbell.
4. ExecuteDMA + enginesH2D stage, analog/digital compute, D2H result; enforce range and direction.
5. CompleteCQ + MSI-XDevice writes completion before interrupt; KMD acknowledges and wakes waiter.
6. ReclaimFence then unmapCPU reads result only after completion/sync; unmap, detach, or retain for reuse.

sq[idx].command_iova = cpu_to_le64(cmd_iova);
sq[idx].command_bytes = cpu_to_le32(cmd_bytes);
sq[idx].user_cookie = cpu_to_le64(cookie);

  dma_rmb();
  complete_submission(le64_to_cpu(cq[idx].user_cookie),
                      le32_to_cpu(cq[idx].status));
}

/* Before reset/teardown, force a posted MMIO write to reach the endpoint. */

readl(adev->bar0 + ACIM_DEVICE_STATUS);

Ordering boundary: coherent memory does not remove the need for barriers. dma_wmb() and dma_rmb() order shared DMA memory; they do not themselves order arbitrary MMIO. Use the kernel MMIO accessors and the hardware protocol's readback rule. MSI/MSI-X is preferred because PCI ordering ensures the device's preceding memory writes arrive before the interrupt-generating write.
