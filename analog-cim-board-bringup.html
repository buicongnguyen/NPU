Terminology map

Define the contract, layer, and measurement behind each term

This glossary explains how each term is used in this ACiM design, not only what its letters expand to. A term may name a source interface, binary contract, compiler representation, transport mechanism, hardware block, numerical assumption, or licensing record; treating those categories as interchangeable creates architecture and implementation errors.

Software interfaces, ownership, and versioned contracts

API
Application Programming InterfaceThe source-level functions, classes, and data types one component exposes. Recompiling can adapt an API change; an already-built binary cannot.
ABI
Application Binary InterfaceThe binary contract for calling conventions, symbol names, data layout, alignment, opcodes, and version compatibility. This design has separate artifact, runtime-UMD, UMD-KMD, and firmware command ABIs.
C ABI
C language binary interfaceA stable function-call boundary using C-compatible symbols and layouts. It is often used for SDK plugins because C++ name mangling and standard-library ABIs vary by toolchain.
UAPI
Userspace API and ABIThe Linux kernel/userspace contract: device files, ioctl numbers, structures, flags, handles, mmap behavior, events, and errors. Once shipped, compatibility must be managed deliberately.
SDK
Software Development KitThe developer-facing product: headers, libraries, compiler, runtime, tools, examples, diagnostics, documentation, packages, and compatibility policy. An SDK is more than a compiler.
HAL
Hardware Abstraction LayerA narrow interface hiding transport details so the same runtime can target a simulator, VFIO prototype, Linux driver, FPGA, or silicon backend.
Runtime
Execution orchestration libraryLoads compiled artifacts, creates sessions, allocates and binds buffers, selects queues, submits work, waits on events, recovers faults, and exposes profiling.
UMD
User-Mode DriverAn unprivileged userspace library that discovers devices, owns process-local objects, prepares descriptors, and translates HAL/runtime requests into the kernel UAPI.
KMD
Kernel-Mode DriverPrivileged kernel code that owns PCIe resources, DMA isolation, interrupts, reset, power, firmware loading, process cleanup, and device lifetime.
Driver
Software controlling a device boundaryAn imprecise word unless qualified. In this guide it can mean UMD, KMD, or both; firmware is a separate device-resident layer.
Firmware
Device-resident control softwareCode running on an embedded controller that boots the accelerator, validates command descriptors, sequences engines, manages health, and reports faults.
Device ABI
Host-to-firmware binary contractThe exact command, descriptor, queue, register, status, and error formats accepted by a firmware/device generation. It is not the model IR.
ISA
Instruction Set ArchitectureThe programmer-visible instructions, registers, address model, and execution rules of a processor or accelerator engine. An ABI may select and constrain an ISA but is not the ISA itself.
Microcode
Low-level engine control programA compact target-specific sequence that drives internal datapaths. It normally sits below compiler IR and above fixed hardware state machines.
Artifact
Deployable compiler outputA versioned package containing code or commands, constants, buffer metadata, target requirements, debug maps, calibration references, and provenance. It is the hinge between offline build and runtime.
Schema
Machine-readable data definitionDefines fields, types, required values, defaults, and evolution rules for an artifact, command, trace, or capability description.
SemVer
Semantic VersioningA major.minor.patch convention. It helps communicate compatibility but does not create compatibility unless each ABI and schema has explicit rules and tests.
SoC
System-on-ChipA chip integrating processors, memory controllers, interconnect, accelerators, security, and peripherals. A PCIe accelerator card may contain one or more SoCs or chiplets.
CI
Continuous IntegrationAutomated build, static analysis, unit, compatibility, simulator, and regression gates run when code or generated contracts change.
CD
Continuous Delivery or DeploymentAutomated packaging and promotion after CI. Driver signing, firmware compatibility, rollback, and hardware qualification make accelerator CD different from a web-only release.
DKMS
Dynamic Kernel Module SupportLinux packaging that rebuilds an out-of-tree kernel module for installed kernels. It eases installation but does not guarantee source compatibility with future kernels.
MVP
Minimum Viable ProductThe smallest end-to-end workflow that proves user value. For this stack, a one-operator vertical slice is an engineering milestone, not necessarily a market MVP.
SLO
Service-Level ObjectiveAn internal measurable target such as p99 latency, availability, or recovery time. Several SLOs may support one customer commitment.
SLA
Service-Level AgreementAn external agreement defining promised service and remedies. A silicon accuracy target or benchmark alone is not an SLA.

Model formats, compiler representations, and transformations

ONNX
Open Neural Network ExchangeA serialized model graph format with typed operators, tensors, attributes, and versioned operator sets. It is an input interchange format, not a hardware schedule.
Opset
Operator-set versionThe ONNX version governing operator semantics. A model's IR version and opset imports are separate compatibility fields and both must be checked.
Graph
Operations connected by data dependenciesThe compiler view of model computation. Dynamic control flow, state, aliases, side effects, and symbolic dimensions make it richer than a simple directed acyclic graph.
Tensor
Typed multidimensional valueDefined by element type, rank, shape, layout, quantization, and memory placement. Equal shapes do not imply equal physical layouts.
IR
Intermediate RepresentationThe compiler's structured program form between source model and executable. Different IR levels preserve different semantics and expose different optimization choices.
HLO
High-Level OperationsA tensor-operation IR family associated with XLA. HLO names a level of representation; it is not one universal file format or device ABI.
StableHLO
Versioned high-level ML operation setA portability layer built on MLIR for carrying framework semantics to compilers with compatibility guarantees. It does not define ACiM placement, calibration, runtime calls, or firmware commands.
MLIR
Multi-Level Intermediate RepresentationCompiler infrastructure for defining dialects and progressively lowering programs. MLIR is the framework; a particular dialect and pass pipeline form the target compiler.
Dialect
MLIR vocabulary and rulesA namespace of operations, types, attributes, parsers, printers, and verification rules for one abstraction level or domain.
Op
Compiler operationAn IR instruction with operands, results, attributes, regions, and semantics. It may represent a whole convolution, a DMA copy, or one target command depending on the dialect.
SSA
Static Single AssignmentAn IR property in which each value is defined once. Dataflow and use-def chains become explicit, enabling analysis and safe rewrites.
DSL
Domain-Specific LanguageA language designed for one domain, such as tensor kernels or schedules. A DSL may lower into MLIR; it is not synonymous with an MLIR dialect, which is an IR extension mechanism.
Pass
Compiler analysis or transformationA staged unit that inspects or rewrites IR. Production pipelines declare ordering, preconditions, invariants, diagnostics, and tests rather than collecting unrelated rewrites.
Canonicalize
Normalize equivalent IR formsRewrites expressions into predictable simpler forms. It reduces pattern count but must preserve semantics and is not target lowering by itself.
Legalize
Replace unsupported constructsTransforms operations, types, shapes, or layouts until every remaining construct is valid for the next dialect or target capability contract.
Lowering
Move to a more concrete abstractionConverts high-level model meaning into target operations, schedules, buffers, commands, or machine code while recording any precision and fallback decisions.
Fusion
Combine operations into one execution regionCan remove intermediate traffic and launch overhead, but may increase live memory, analog range, converter work, or unsupported-op pressure.
Partition
Assign graph regions to execution backendsSeparates ACiM-supported work from DSP, CPU, or other fallback work. Boundary conversion and transfer costs belong in the decision.
Frontend
Source-facing compiler stageImports ONNX, StableHLO, or framework programs, validates semantics, normalizes shapes and types, and emits a compiler IR.
Backend
Target-facing compiler stageMaps legal IR to one device's arrays, memories, converters, command format, code, and artifact ABI. This is the least reusable part across Tenstorrent and ACiM.
Codegen
Code or command generationEmits machine instructions, microcode, command streams, constants, relocations, and metadata after scheduling and memory planning.
AOT
Ahead-of-Time compilationCompiles before deployment, enabling qualification and caching. It requires a clear target descriptor and artifact compatibility contract.
JIT
Just-in-Time compilationCompiles at or near execution time for actual shapes or hardware. It improves specialization but adds latency, security, reproducibility, and cache concerns.
ODS
Operation Definition SpecificationMLIR's declarative way to define operations, types, and attributes using TableGen records, from which C++ declarations, verifiers, builders, and documentation can be generated.
TableGen
LLVM declarative record language and generatorDescribes structured compiler data and emits repetitive C++ or documentation. It is a build-time generator, not the runtime artifact schema.
LLVM IR
LLVM low-level intermediate representationA typed SSA representation suited to CPU-like code generation. Tensor and ACiM array semantics usually need higher-level dialects before lowering toward LLVM IR.
TOSA
Tensor Operator Set ArchitectureAn Arm-defined tensor operator specification and MLIR dialect intended as a common layer between frameworks and hardware-oriented lowering.
TVM
Apache TVMAn end-to-end machine-learning compiler stack with graph and tensor scheduling abstractions. Its architecture can be studied without assuming its cost model fits analog arrays.
PJRT
OpenXLA uniform device APIA C API and plugin boundary through which frameworks access opaque, device-specific implementations. It is neither a model IR nor a kernel driver UAPI.
ORT
ONNX RuntimeA model runtime that can partition supported graph regions to a hardware Execution Provider while retaining other regions on fallback providers.
EP
Execution ProviderAn ONNX Runtime plugin that advertises capability, compiles claimed subgraphs, creates runtime state, and invokes the accelerator.
Target
Target descriptorA machine-readable description of device generation, capacities, supported formats, topology, firmware/ABI versions, errata, and calibrated capabilities used by the compiler.
Layout
Logical-to-physical tensor arrangementDefines dimension order, blocking, strides, sharding, bank placement, and padding. Layout conversions can dominate a theoretically cheap operator.

Host/device transport, addressing, queues, and interrupts

PCIe
Peripheral Component Interconnect ExpressThe host/device interconnect carrying configuration, MMIO, DMA transactions, and message-signaled interrupts.
BAR
Base Address RegisterPCIe-discovered address region exposing device control registers or a device-memory aperture to the host.
MMIO
Memory-Mapped Input/OutputCPU loads and stores directed to device registers rather than ordinary RAM.
PIO
Programmed Input/OutputThe CPU directly reads or writes device registers/data instead of asking a DMA engine to transfer a buffer.
DMA
Direct Memory AccessDevice-initiated transfer to or from host memory without the CPU copying each byte.
IOMMU
Input-Output Memory Management UnitTranslates and restricts device DMA addresses so a context cannot access arbitrary host memory.
IOVA
Input-Output Virtual AddressThe address a device places in a DMA transaction after the kernel establishes an IOMMU/DMA mapping.
BO
Buffer ObjectDriver-managed allocation or imported buffer represented to userspace by an opaque handle.
Descriptor
Fixed-format work or transfer recordContains buffer addresses, sizes, operation identifiers, dependencies, flags, and a cookie. Its byte layout is part of the device ABI.
SQ
Submission QueueRing where the host publishes validated commands for firmware/device consumption.
CQ
Completion QueueRing where the device reports terminal status, fault information, counters, and the matching cookie.
Doorbell
MMIO notification registerA host write telling the device that a queue producer index has advanced.
IRQ
Interrupt RequestKernel-visible notification that completion, fault, firmware, or health work needs service.
MSI-X
Message Signaled Interrupts ExtendedPCIe interrupt mechanism supporting multiple independently configured vectors.
FLR
Function Level ResetPCIe reset of one function without resetting the entire machine or unrelated functions.
AER
Advanced Error ReportingPCIe facility reporting correctable, non-fatal, and fatal link/transaction errors.
FD
File DescriptorPer-process Linux handle used for devices, dma-buf objects, events, and other kernel resources.
dma-buf
Shared DMA buffer frameworkLinux mechanism for exporting and importing the same buffer across device drivers.
RO/RW/WO
Register access modesRead-only, read/write, and write-only. RW1C means writing one clears the selected status bit.
DRM
Direct Rendering ManagerLinux kernel subsystem reused by drivers/accel; here DRM does not mean digital-rights management.
TLB
Translation Lookaside BufferHardware cache of address translations; a vendor-specific TLB programming model is not portable by default.
SMMU
System Memory Management UnitArm's I/O memory-management architecture used to translate and isolate device accesses.
NUMA
Non-Uniform Memory AccessHost architecture where memory latency/bandwidth depends on which CPU and memory node are used.
VFIO
Virtual Function I/OLinux framework for IOMMU-protected direct device access from a trusted userspace driver. Useful for prototypes, but production ownership and sharing still need design.
UIO
Userspace I/OA small Linux framework exposing simple MMIO and interrupts. It does not provide VFIO-style IOMMU isolation or a full accelerator memory model.
ioctl
Linux device-control system callTransfers a numbered request and fixed UAPI structure between userspace and the KMD. Pointer size, padding, bounds, and backward compatibility require explicit design.
Fence
Asynchronous completion dependencyA waitable object indicating that prior work has completed. It orders resource reuse; it is different from a CPU or DMA memory barrier.
SG
Scatter-gather listA sequence of physical or DMA segments representing one logical buffer so an engine can transfer non-contiguous memory without a CPU copy.

Digital compute, on-chip interconnect, memory, and implementation

NPU
Neural Processing UnitAn accelerator specialized for neural-network operations. It may be digital, analog, or hybrid; the name alone does not identify its arithmetic or memory architecture.
HWA
Hardware AcceleratorA specialized engine offloading work from a general CPU. NPU, DSP, video codec, DMA engine, and ACiM array are all possible HWAs.
NoC
Network-on-ChipThe internal packet or circuit-switched fabric connecting cores, tiles, SRAM banks, DMA engines, converters, and controllers. It is inside the device, unlike the PCIe host link.
Tile
Repeated compute and memory unitA physical or logical building block containing some combination of array, digital core, local SRAM, router, converters, and control. Its contents are architecture-specific.
Core
Programmable execution engineA processor that fetches instructions and manipulates state. An analog array performs MVM physics but is not itself a RISC-V-like core.
RISC-V
Open instruction-set architectureOften used for embedded control, scheduling, scalar work, and firmware. Too much control work can saturate cores and the NoC even when arrays are underused.
CP
Command ProcessorA device block or firmware core that fetches descriptors, checks dependencies, configures engines, and publishes completions.
Control plane
Lifecycle and policy pathOwns discovery, allocation, security, power, reset, health, version negotiation, and observability rather than tensor payload movement.
Data plane
Tensor and command execution pathCarries inputs, outputs, weights, partial sums, commands, and completion state through the hot path.
DSP
Digital Signal ProcessorA programmable digital engine efficient for vector, activation, reduction, preprocessing, and unsupported operators surrounding the analog MVM path.
SIMD
Single Instruction, Multiple DataOne instruction applies the same operation to several lanes. It describes data parallelism, not the complete processor schedule.
VLIW
Very Long Instruction WordAn ISA style encoding several operations for parallel issue, with much scheduling responsibility placed on the compiler.
SRAM
Static Random-Access MemoryFast volatile on-chip memory used for activations, commands, scratch data, and partial sums. It is distinct from non-volatile analog weight storage.
DRAM
Dynamic Random-Access MemoryHigh-capacity volatile memory, usually off-chip, holding models and tensors that do not fit locally. Moving data to and from DRAM is often expensive.
L1
Level-one local memory or cacheThe nearest programmable storage to a core or tile. In accelerator documentation, confirm whether L1 is software-managed SRAM, a cache, or both.
Cache
Hardware-managed copy of memoryExploits locality and is usually transparent to software. Scratchpad SRAM is explicitly allocated and moved by compiler/runtime, so it is not automatically a cache.
Double buffer
Overlap transfer and compute with two buffersWhile one buffer feeds compute, the other is filled or drained. It hides latency only when bandwidth, dependencies, and engine concurrency permit overlap.
ASIC
Application-Specific Integrated CircuitFixed-function or specialized silicon manufactured for a target application, unlike a reprogrammable FPGA.
RTL
Register-Transfer LevelA clocked digital hardware description of registers and transfers used to implement and verify controllers, buses, DMA, NoC, and digital compute.
SystemC
C++ system-modeling libraryUsed for functional, transaction-level, or cycle-oriented models before RTL and silicon. Model timing accuracy must be stated rather than inferred from the language.
FPGA
Field-Programmable Gate ArrayReconfigurable hardware used to prototype PCIe, firmware, commands, and digital engines before ASIC availability; it does not reproduce analog array physics by itself.
ROM
Read-Only MemoryImmutable or protected first-stage device code/data establishing reset-safe boot, identity, and firmware loading.
PPA
Power, Performance, and AreaThree coupled hardware implementation objectives. Improving array arithmetic alone may worsen converters, control, memory, or die area.
DFT
Design for TestStructures and methods that make manufactured silicon observable and testable, including scan, memory test, analog test modes, and calibration access.
BIST
Built-In Self-TestOn-device logic or firmware that runs known tests for memories, links, arrays, converters, or controllers and reports health.
Bandwidth
Data transferred per unit timeA capacity such as GB/s. Peak link bandwidth, sustained payload bandwidth, NoC bisection bandwidth, and per-array bandwidth are different measurements.
Latency
Time for one operation or requestState the scope and percentile: kernel, layer, model, compile, first-token, per-token, median, or p99.
Throughput
Completed work per unit timeExamples are inferences/s, tokens/s, or frames/s. High batching can improve throughput while worsening single-request latency.
TOPS
Tera Operations Per SecondA peak or measured arithmetic rate. The operation-count convention, precision, sparsity, utilization, and included digital/converter work must be disclosed.
TOPS/W
Operations per second per wattAn efficiency ratio whose value depends on operation counting and power boundary. Array-only peak TOPS/W is not end-to-end application efficiency.
H2D
Host-to-Device transferInputs, commands, constants, or firmware moving from host memory to the accelerator.
D2H
Device-to-Host transferOutputs, traces, or status moving from the accelerator to host memory.
SKU
Stock Keeping UnitA sellable board or device configuration with a defined chip count, memory, power envelope, firmware support, and qualification status.

Boot, system images, simulation, and low-level debug

QEMU
Machine emulator and virtualizerSystem emulation runs a guest CPU, memory, and only the devices implemented by the selected QEMU machine. A generic Arm or RISC-V virt board does not automatically contain an ACiM endpoint.
BSP
Board Support PackageThe bootloader, kernel configuration, device tree, drivers, firmware, root-filesystem integration, and build metadata needed to boot a specific board.
Bootloader
Software that starts the host operating systemInitializes enough platform hardware to select and load the host Linux kernel, device tree, and optional initramfs. It is separate from ACiM controller firmware.
Rootfs
Root filesystemThe Linux userspace image containing libraries, configuration, UMD, runtime, tools, services, and applications. An initramfs is a rootfs held in memory during early boot.
Cross-compile
Build on one architecture for anotherFor example, build Arm64 Linux userspace on an x86 workstation or RISC-V controller firmware with a freestanding target toolchain and linker script.
JTAG
Board-level debug and programming interfaceLets a debugger halt a supported target, inspect registers and memory, and load or flash images through a tool such as OpenOCD. It is a bring-up path, not the normal application transport.
remoteproc
Linux remote-processor frameworkPlatform mechanism for loading firmware, powering, starting, stopping, and reporting crashes for an on-SoC remote core. It is relevant when the ACiM controller is a heterogeneous SoC processor rather than a PCIe card controller.
Co-simulation
Several models executing one protocolA host or QEMU machine model forwards MMIO, DMA, queues, and interrupts to a functional, cycle, RTL, or analog model so the complete software chain runs before silicon.
ELF
Executable and Linkable FormatA structured binary containing loadable segments, symbols, relocations, and an entry point. Linux executables, kernel modules, and controller firmware may all use ELF but target different machines and loaders.
DTB
Device Tree BlobBinary hardware description passed to Linux on many embedded systems. It declares platform devices, MMIO ranges, interrupts, clocks, resets, IOMMUs, reserved memory, and relationships such as remote processors.

Analog CIM, numerical formats, model execution, and accuracy

ACiM
Analog Compute-in-MemoryPerforms arithmetic through analog behavior inside or adjacent to memory arrays to reduce weight movement. AiCM, AIMC, analog CIM, and analog in-memory computing are overlapping spellings.
MVM
Matrix-Vector MultiplicationComputes y = Wx for one vector. In ACiM, weights map to cell conductances, inputs drive rows, column currents sum products, and converters plus digital reconstruction finish the operation.
VMM
Vector-Matrix MultiplicationOften used in analog literature for the same array operation viewed with a row-vector convention. Confirm tensor orientation instead of assuming VMM and MVM imply different hardware.
GEMM
General Matrix MultiplicationComputes C = alpha AB + beta C for matrices. A compiler normally decomposes GEMM into tiled MVMs, batches, partial sums, and digital accumulation.
MAC
Multiply-AccumulateMultiplies two values and adds the product into an accumulator. Marketing may count one MAC as one or two operations, so TOPS comparisons need a declared convention.
Crossbar
Intersecting row and column arrayMemory cells at intersections encode weights; driven rows and sensed columns exploit Ohm's and Kirchhoff's laws for parallel multiply and accumulation.
WL / BL
Wordline and bitlineRow and column conductors used to select, drive, program, and sense memory cells. Parasitic resistance and capacitance make position and array size matter.
Conductance
Electrical weight representationThe inverse of resistance, measured in siemens. Devices encode a weight through one or more conductance states, often using differential cells for signed values.
DAC
Digital-to-Analog ConverterConverts encoded digital inputs into voltage, current, pulse width, or bit-serial drive for an array. Some designs replace a conventional multibit DAC with simpler drivers and more cycles.
ADC
Analog-to-Digital ConverterDigitizes analog column results. Resolution, sampling rate, sharing, range, and calibration can dominate ACiM area, energy, latency, and accuracy.
ENOB
Effective Number of BitsThe useful converter resolution after measured noise and distortion. It is commonly lower than the nominal ADC bit count and varies with speed, range, and PVT.
SNR
Signal-to-Noise RatioThe ratio of desired signal power to noise power, usually in decibels. SNR helps characterize a path but does not directly equal model accuracy.
Partial sum
Incomplete dot-product resultProduced when a tensor is split across rows, columns, bit slices, arrays, or chiplets. It must be accumulated with declared precision and placement.
NVM
Non-Volatile MemoryRetains state without power. Flash, RRAM, and PCM are different NVM technologies with different programming, variation, endurance, density, and sensing behavior.
NOR / NAND
Flash memory organizationsNOR favors random access and simpler compute mapping; NAND favors density but uses string/page organization and more complex access. A roadmap claim does not make their ACiM backends interchangeable.
RRAM
Resistive Random-Access MemoryAlso called ReRAM; stores data through programmable resistance. Analog levels face device variability, nonlinear programming, drift, defects, and endurance limits.
PCM
Phase-Change MemoryStores state through material phases with different resistance. Multi-level programming, drift, write asymmetry, and temperature sensitivity shape its compute use.
Write-verify
Closed-loop cell programmingApplies a programming pulse, reads the cell, compares against a target, and repeats. It improves weight placement but costs time, energy, and endurance.
Calibration
Measured correction of hardware behaviorEstimates converter offsets, gains, array transfer functions, drift, defects, or per-tile scales and stores parameters used by compiler, firmware, or runtime.
Drift
Time-dependent state changeStored conductance or circuit response moves after programming. Monitoring, compensation, refresh, retraining, or fallback must be included in lifecycle cost.
PVT
Process, Voltage, and TemperatureManufacturing and operating conditions that change timing, analog transfer behavior, converter quality, retention, and calibration validity.
SPICE
Transistor-level circuit simulationModels electrical behavior and corners at high fidelity but limited scale. It complements, rather than replaces, architectural and task-level simulation.
Quantization
Map values to a finite set of levelsDefines precision, scale, zero point, rounding, clipping, and sometimes per-channel parameters. Analog error is additional to digital quantization error.
INT8 / INT4
8-bit and 4-bit integer formatsDescribe logical tensor values, not automatically ADC ENOB or cell bits. A physical implementation may use several cells, slices, reads, or accumulations per logical value.
ANA8
Vendor analog-precision labelNot an industry-standard numeric type. Treat it as a product claim until input encoding, weight representation, accumulation, ENOB, calibration, and validation are specified.
FP32 / BF16
Floating-point formatsFP32 has 32 bits; bfloat16 keeps an FP32-like exponent with fewer fraction bits. A floating-point baseline must be named when reporting quantized or analog accuracy loss.
PTQ
Post-Training QuantizationConverts a trained model using calibration data without full retraining. It is fast but may struggle with sensitive layers, outliers, or analog nonidealities.
QAT
Quantization-Aware TrainingModels quantization effects during training or fine-tuning so weights adapt. Hardware-aware training may additionally inject measured analog behavior.
QDQ
Quantize/Dequantize operatorsGraph nodes that make scale, zero point, rounding, and precision transitions explicit. They describe numerical intent, not necessarily physical converter operations.
Scale / ZP
Quantization scale and zero pointMap real values to integer codes. Scale controls step size; zero point represents real zero in an asymmetric integer range.
Clip / saturate
Limit values to a representable rangeClipping is the chosen range restriction; saturation is output behavior at the limits. Both can create systematic model error and must be traced per layer.
ECC
Error-Correcting CodeRedundant coding that detects or corrects digital bit corruption. It does not generally correct continuous analog compute error or converter nonlinearity.
YOLO
You Only Look OnceFamily of single-stage object-detection models used here as the first end-to-end workload.
NMS
Non-Maximum SuppressionPostprocessing that removes redundant overlapping detections.
mAP
mean Average PrecisionObject-detection accuracy metric computed over classes and intersection-over-union criteria.
LLM
Large Language ModelTransformer model with large parameter and runtime-state requirements; referenced by the companion scale-out design.
KV cache
Transformer key/value cacheRuntime state retained for prior tokens so attention need not recompute them. Its capacity, bandwidth, update pattern, and placement differ from static model weights.
Attention
Content-dependent token mixingTransformer operation using query, key, and value tensors. Linear projections fit MVM arrays; softmax, dynamic KV access, reductions, and communication need other engines.
Batch
Requests processed togetherBatching can raise array utilization and throughput but consumes memory and may increase queueing and tail latency.
p95 / p99
Tail-latency percentilesThe time below which 95% or 99% of requests complete. They expose contention, calibration, fallback, and recovery stalls hidden by an average.
FPS
Frames Per SecondSteady-state image/video throughput. It must be paired with batch size, latency distribution, power, and task accuracy.

Open-source licensing, software inventory, and provenance

SPDX
Software Package Data ExchangeAn open standard for machine-readable software, component, license, security, and provenance information. SPDX records facts; it does not grant rights or replace license text and legal review.
SBOM
Software Bill of MaterialsAn inventory of components and relationships in a product or build. SPDX is one format for an SBOM, but an SPDX file may also describe other lifecycle information.
SPDX ID
SPDX license identifierA standardized short name such as Apache-2.0. A source header like SPDX-License-Identifier: Apache-2.0 labels the applicable license.
Expression
SPDX license expressionA machine-readable combination using operators such as AND, OR, and WITH. The operator changes obligations and choice, so it is not decorative metadata.
Permissive
License family with limited downstream conditionsCommon examples include Apache-2.0, MIT, and BSD licenses. Exact notice, attribution, patent, and redistribution terms still differ.
Copyleft
License conditions tied to redistribution or modificationSome licenses require corresponding source or same-license terms in defined circumstances. Scope varies, so architecture labels are not a legal conclusion.
Apache-2.0
Apache License 2.0A permissive license with copyright, notice, and express patent-license provisions. Preserve required notices and inspect separately licensed dependencies and assets.
MIT / BSD
Permissive license familiesGenerally short licenses centered on retaining copyright and permission notices, but variants differ and must be identified precisely.
GPL / LGPL
GNU copyleft license familiesGPL and Lesser GPL have different conditions and version choices. Kernel modules, userspace libraries, linking, modification, and distribution require component-specific review.
NOTICE
Distribution notice fileMay collect attribution required by Apache-licensed or third-party material. It does not replace the LICENSE file and should not be discarded during packaging.
Provenance
Origin and transformation historyRecords repository, commit, author or supplier, license, patches, generator, model/dataset source, and build inputs so a shipped artifact can be reproduced and audited.
Transitive dep.
Dependency of a dependencyCode or data pulled indirectly into a build. Top-level repository licensing does not automatically describe the full dependency closure.
Upstream
Original or authoritative project lineContributing reusable changes upstream reduces long-term divergence, but product schedules and target confidentiality may require a maintained downstream boundary.
Fork / vendor

Do not confuse

Similar words describe different contracts

API vs ABI vs UAPIAn API is what source code calls. An ABI is the binary representation and behavior. A UAPI is specifically the kernel/userspace API and ABI surface.
Runtime vs UMD vs KMD vs firmwareThe runtime schedules model execution; the UMD translates userspace objects; the KMD enforces host protection; firmware controls device engines.
ONNX vs StableHLO vs MLIRONNX is a model interchange format. StableHLO is a versioned high-level tensor operation set carried in MLIR. MLIR is the infrastructure used to define many IR dialects and lowering stages.
DSL vs MLIR dialectA DSL is a user- or developer-facing language specialized for a domain. A dialect is a structured extension of MLIR; a DSL can compile into one or several dialects.
ABI vs ISA vs microcodeAn ABI defines how binaries and components interact. An ISA defines processor-visible instructions and state. Microcode is a low-level program or control encoding for a particular engine.
BAR/MMIO vs DMABAR-backed MMIO carries small control/register accesses. DMA moves bulk buffers between host memory and the device.
PCIe vs DMA vs NoCPCIe is the host link. DMA is an engine/protocol for moving buffers over an addressable path. The NoC is the internal device fabric connecting tiles and memories.
SQ/CQ vs doorbell/MSI-XSQ and CQ hold durable command/completion records. A doorbell and MSI-X only notify the other side that records are ready.
IOVA vs CPU pointer vs physical addressA CPU pointer belongs to a process, a physical address names RAM, and an IOVA is the device-visible address produced by the DMA/IOMMU mapping.
MVM vs GEMM vs MACA MAC is one arithmetic primitive. MVM multiplies a matrix by one vector. GEMM multiplies matrices and is tiled into many MVM-like operations plus partial-sum accumulation.
Nominal ADC bits vs ENOBNominal bits describe output code width. ENOB describes useful measured resolution after noise and distortion; it usually changes with sample rate, range, and PVT.
Bandwidth vs latency vs throughputBandwidth is transfer capacity, latency is time for one request, and throughput is completed work per time. Improving one does not guarantee improvement in the others.
Quantization error vs transport errorADC, low precision, and calibration affect numerical accuracy. PCIe AER, DMA, ECC, and queue faults affect reliable delivery and execution.
SPDX vs SBOM vs LICENSE/NOTICESPDX is a machine-readable standard; an SBOM is a component inventory; LICENSE and NOTICE carry legal text and attribution. None substitutes automatically for the others.

Driver decision

Keep policy, graph execution, artifact parsing, and batching in a UMD/runtime. Whole-chain verification

Test one contract at a time before asking the complete board to run a model

QEMU is useful, but it is not an analog Compute-in-Memory (CIM) simulator by default. Use it to boot a supported host or controller CPU and its software. Add a custom device model, fake transport, or co-simulation bridge when the Kernel-Mode Driver (KMD), firmware, queue protocol, and analog model must run together.

Terms here BSP = bootloader, kernel, device tree, and board support rootfs = Linux userspace files HAL = transport-neutral device interface JTAG = low-level hardware debug port Co-simulation = coordinated execution of two or more simulator models
Direct answer There is no universal "QEMU, SD card, then device kernel" sequence Choose the deployment topology first; build, load, and debug each artifact through its actual owner.

A desktop PCIe card normally needs no SD card: Linux is already running, the KMD loads controller firmware when the device has a programmable controller, and the runtime submits a compiled model artifact. An embedded host may boot a BSP image from SD/eMMC and then load accelerator firmware, when present, through remoteproc or a vendor driver. A device compute kernel or microcode payload is not the Linux kernel, and should be named separately.

Proposed Linux PCI probe pathEnable, claim, map, configure DMA, IRQ, firmware, then expose userspace
static int acim_probe(struct pci_dev *pdev,
                      const struct pci_device_id *id)
{
  struct acim_dev *adev;
  const struct firmware *fw = NULL;
  int ret, i;

  ret = pcim_enable_device(pdev);
  if (ret)
    return ret;

  ret = pcim_iomap_regions(pdev, BIT(0), "acim");
  if (ret)
    return ret;

  adev = devm_kzalloc(&pdev->dev, sizeof(*adev), GFP_KERNEL);
  if (!adev)
    return -ENOMEM;

  adev->bar0 = pcim_iomap_table(pdev)[0];
  pci_set_drvdata(pdev, adev);

  ret = acim_read_and_validate_caps(adev);  /* ROM-visible; no DMA yet */
  if (ret)
    return ret;

  if (adev->caps.dma_bits != 32 && adev->caps.dma_bits != 64)
    return -EPROTO;

  ret = dma_set_mask_and_coherent(&pdev->dev,
                                  DMA_BIT_MASK(adev->caps.dma_bits));
  if (ret)
    return dev_err_probe(&pdev->dev, ret, "unsupported DMA mask\n");

  pci_set_master(pdev);                 /* device may now initiate DMA */

  ret = acim_alloc_admin_rings(adev);       /* dma_alloc_coherent */
  if (ret)
    goto err_clear_master;

  ret = pci_alloc_irq_vectors(pdev, 1, adev->caps.max_vectors,
                              PCI_IRQ_MSI | PCI_IRQ_MSIX);
  if (ret < 0)
    goto err_free_rings;
  adev->nr_vectors = ret;

  for (i = 0; i < adev->nr_vectors; ++i) {
    ret = request_threaded_irq(pci_irq_vector(pdev, i),
                               acim_irq, acim_irq_thread, 0,
                               "acim", adev);
    if (ret)
      goto err_free_irqs;
  }

  ret = request_firmware(&fw, adev->caps.firmware_name, &pdev->dev);
  if (ret)
    goto err_free_irqs;

  ret = acim_boot_firmware(adev, fw);       /* reset, load, hash, start, ABI */
  release_firmware(fw);
  if (ret)
    goto err_free_irqs;

  ret = acim_register_accel_or_chardev(adev); /* userspace opens only now */
  if (ret)
    goto err_quiesce;

  return 0;

err_quiesce:
  acim_quiesce_and_reset(adev);
err_free_irqs:
  while (i--)
    free_irq(pci_irq_vector(pdev, i), adev);
err_free_vectors:
  pci_free_irq_vectors(pdev);
err_free_rings:
  acim_free_admin_rings(adev);
err_clear_master:
  pci_clear_master(pdev);
  return ret;
}
1. BlockStop new opens/submitsMark device removing or lost; wake waiters with a terminal error.
2. QuiesceMask and drainDisable queues, stop engines and DMA, synchronize IRQ handlers.
3. ReclaimFinish ownershipSignal/cancel fences, unmap streaming DMA, detach dma-buf and unpin pages.
4. ReleaseIRQ and ringsFree vectors, coherent control memory, firmware state and device node.
5. DisableClear bus masterConfirm no bus transaction remains before managed BAR/device release.

A failed firmware boot is a detected device that is not ready for userspace.

Implementation repository

The skeleton is reasonable; make code, artifacts, data, and loaders traceable

A directory tree is only the first layer of the design. Every owned source path should produce a named build target or validated data file, identify who loads it, state how long it remains valid, and name the smallest test that proves the boundary. The separate repository blueprint covers CMake, compilers, CI, GoogleTest, and contribution mechanics in greater depth; this section maps those mechanics onto ACiM board boot and model execution.

Staged delivery

Reach the board through nine evidence-producing milestones

Each milestone is a vertical slice with owned code, a reproducible operation, named boundary data, a measurable exit gate, and an explicit capability that it unlocks. Transport semantics remain common while an integrated SoC uses MMIO/DMA/SMMU/IRQ and a PCIe device uses BAR/DMA/IOMMU/MSI-X.

Terms here Protocol v0 = first versioned hardware/software contract FPGA = Field-Programmable Gate Array prototype MVM = Matrix-Vector Multiplication Exit gate = evidence required before depending on this milestone Unlock = new work that becomes technically meaningful after the gate passes Qualification = evidence required before product release

Two plans with different jobs: the ten-stage verification ladder answers where and at what fidelity a test runs, from native model through silicon. These nine delivery milestones answer which dependency-ordered code slice the team implements. For example, the queue slice should pass first in the fake backend, then QEMU/co-simulation, FPGA, and finally silicon without changing its descriptor semantics or whether the physical transport is integrated or PCIe.
