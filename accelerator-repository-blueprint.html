Accelerator Repository Blueprint

Build a repository that can grow from one simulated matrix operation into a maintained compiler, runtime, SDK, firmware, driver, model library, and hardware qualification system. TT-Metal supplies the concrete reference; this page extracts the boundaries and test discipline that also apply to an Analog CIM machine.

Observed in TT-Metal

The repository is an engineering system, not a large source folder

The current TT-Metal repository separates the Metalium programming layer, TT-NN, models, tests, documentation, tools, scripts, build infrastructure, and GitHub automation. The exact names and scale are Tenstorrent-specific; the separation of ownership is the useful pattern.

The current test tree is purpose-oriented: tests/tt_metal/ covers low-level subsystems and microbenchmarks; tests/ttnn/ includes unit, integration, performance, profiling, statistical, stress, and nightly paths; tests/end_to_end_tests/ crosses stack boundaries; tests/sweep_framework/ explores parameter spaces; and tests/nightly/ groups deeper hardware configurations. This is the model for the proposed test ladder later on this page.

Copy the architecture of responsibility, not the repository size. A new ACiM stack should begin with one contract path, one simulator backend, one runtime path, and one operator. Create a directory only when it has a clear owner, dependency direction, test lane, and generated artifact.

Repository terminology

Terms used throughout the blueprint

ABI: Application Binary InterfaceThe binary-level contract for calling functions or exchanging structures across independently built components.
HAL: Hardware Abstraction LayerAn internal device-neutral interface implemented by simulator, fake-device, and real-hardware backends.
SDK: Software Development KitThe supported APIs, libraries, tools, examples, documentation, packages, and diagnostics used by application developers.
UMD / KMDUser-Mode Driver and Kernel-Mode Driver. UMD integrates the runtime; KMD owns privileged OS resources such as DMA, interrupts, isolation and reset.
CI: Continuous IntegrationAutomated build, static analysis, test, packaging, and evidence collection triggered by code changes or schedules.
CTestCMake's test runner and test registry. It can launch GoogleTest executables and label or group native test targets.
Smoke testThe shortest complete health path: discover, open, execute one known operation, compare, close, and leave the device usable.
Sweep testA parameterized exploration across shapes, dtypes, layouts, precision, faults, calibration, or other supported dimensions.
Soak testA long-running stress test intended to reveal leaks, state accumulation, thermal behavior, rare races, and recovery failures.
Golden testA deterministic comparison against an approved IR, artifact, command sequence, tensor, or report. Updates require an explained semantic change.
PVT: Process, Voltage, TemperatureThe manufacturing and operating corners used to characterize timing, power, numerical variation, and reliability.
ADR: Architecture Decision RecordA short document preserving the context, alternatives, chosen design, consequences, migration, tests, and owner of a durable decision.
SKU: Stock Keeping UnitA specific product or board configuration that may differ in devices, memory, topology, clocks, power, firmware, or supported features.
SanitizerCompiler instrumentation such as AddressSanitizer or UndefinedBehaviorSanitizer that detects classes of memory and language errors at runtime.
TracyA real-time frame profiler used here for host C++ zones, frames, messages, plots, threads, locks, memory, and operating-system activity.
ZoneA named interval with a begin and end timestamp. Nested zones divide a host request or device command into attributable phases.
Device cycle counterA monotonically increasing hardware timer read by firmware. Its clock domain and frequency must accompany the captured value.
Clock domainLogic driven by one clock source. Host, controller, array sequencer, converter, DSP, and multiple devices may not share a timestamp origin or rate.
Correlation IDA stable identifier carried through API call, queue command, firmware descriptor, device trace, interrupt, and result so records from different planes can be joined.
Probe effectThe latency, memory, bandwidth, scheduling, or power change caused by observation itself. Profiled and official uninstrumented measurements must be compared.

Rule 1One contract sourceschemas/ generates C++, Python, and firmware readers. Never maintain three handwritten definitions of a command or artifact.
Rule 2Public API is narrowinclude/acim/ exposes supported interfaces. Private implementation headers remain below src/.
Rule 3Hardware is a backendThe runtime targets a HAL implemented by functional simulation, queued fake device, and hardware. Compiler tests do not require a board.
Rule 4Tests state their scopeA developer can tell whether a test needs no device, one device, a topology, calibration data, or a qualification lab from its path and markers.
Rule 5Models are consumersModel demos may use the public SDK; they must not reach into compiler internals, physical addresses, or private driver APIs.
Rule 6Tools consume stable evidenceInspectors and profilers read versioned artifacts and traces. They do not scrape human log text as their primary interface.
Rule 7Configuration is validated dataTarget, test-matrix, model, and threshold files have schemas, units, version fields, examples, and validation tests.
Rule 8Generated output is reproducibleGenerated files record generator version and source hash; CI fails when committed output differs from regeneration.

Dependency direction

The directory tree matters only when dependencies follow it

Keep stable data contracts at the bottom, device-neutral behavior in the middle, and user-facing integrations at the top. Hardware implementations satisfy interfaces; they do not redefine them.

Native build toolchain

Turn one ACiM source file into a repeatable binary and test result

A build stack is a chain of different tools. CMake describes targets and generates a build graph; Ninja or Make executes that graph; GCC, Clang, or MSVC compiles C/C++; an archiver and linker create libraries and executables; CTest runs the native tests. Treating all of this as "the compiler" makes failures difficult to locate.

1. SelectPreset and toolchainHost OS, compiler, build type, features, target/sysroot
2. ConfigureCMakeDetect capabilities, resolve packages, create explicit targets
3. GenerateNinja build graphCommands, dependencies, generated files and output paths
4. CompileGCC / Clang / MSVCPreprocess, parse, optimize, generate objects and diagnostics
5. LinkArchiver and linkerStatic/shared libraries, executables, symbols, ABI and runtime paths
6. ProveCTest and installUnit, smoke, labels, package layout and saved build manifest

A working minimal build is included

ABI rule: record compiler ID/version, target triple, C++ standard library, build type, sanitizer mode and public ABI version. Implementation chain

Build one executable path before expanding the surface area

Each milestone adds exactly one new responsibility and must leave a testable artifact behind. This sequence prevents the model zoo, framework plugin, and hardware backend from hiding an unstable compiler/runtime contract.

Verification architecture

The test chain grows in cost, realism, and diagnostic distance

Lower levels answer narrow questions quickly. Higher levels validate integration and product behavior, but they should never be the first place a compiler rule, allocator invariant, or command encoding is tested.

Performance discipline

A performance test is a controlled experiment, not a timed unit test

Pin the workloadModel/operator hash, shape, batch, sequence length, precision, layout, compiler options and fallback policy.
Pin the machineBoard SKU, device serial, firmware, clocks, power mode, temperature window, target descriptor, calibration and defect-map IDs.
Separate phasesReport compile, weight program, load, warmup and steady-state execution separately; do not hide setup in token or frame throughput.
Use distributionsRecord sample count, warmup count, p50, p95, variability and outliers. One best run is not a regression baseline.
Count causesCycles, bytes, queue wait, DMA, array active time, ADC occupancy, digital fallback, stalls, saturation and retries explain the headline metric.
Version thresholdsStore absolute limits and allowed regression percentages with owner, reason, baseline commit and expiry/review date.
Prove correctness firstA faster incorrect result fails. Associate accuracy output and numerical-policy identity with every accepted performance record.
Control instrumentationUse detailed traces to diagnose, then measure the official result with the declared low-overhead counter set.

ACiM-specific rule: report cold programming, warm resident-weight inference, recalibration, and degraded/remapped execution as different scenarios. A resident-weight MVM number cannot stand in for the latency or energy of deployment, model switching, digital fallback, or recovery.

Tracy and device profiling

Use Tracy directly on the host and preserve device time at the source

Tenstorrent is a useful reference because its tooling exposes both host work and device-side RISC-V work. The reusable architecture is a split design: normal Tracy instrumentation for code running on the host, plus bounded cycle-stamped records for bare-metal firmware and accelerator engines. A clock-correlation and export layer brings the evidence together without changing when the device work appears to have happened.

Recommended ACiM design: use upstream Tracy for host C++ and Python-visible runtime phases; use one fixed-record trace buffer per device producer; drain through the UMD/KMD path; fit each device clock to the host monotonic clock; then export an accurate device trace. Put device intervals into the Tracy UI only through a maintained bridge that accepts their original timestamps.

The current TT-Metalium Tracy guide covers host C++, Python, and device views. The more detailed device program profiler guide shows the device mechanism: kernels emit bounded records, and the host runtime drains them for analysis. These are related views, but not the same instrumentation path.

Concurrency rule: make each device buffer single-producer so the hot emit path needs no shared atomic reservation. For live profiling, use two buffers with explicit device-owned, ready-for-host, host-owned, and free states. The device never writes a host-owned buffer; the host never decodes a device-owned buffer.

Correlate clocks instead of assuming they are synchronized

For sync sample i, record host send t0[i], device cycle c[i], and host receive t1[i]. Approximate the host time of the device read as m[i] = t0[i] + (t1[i] - t0[i]) / 2, then fit host_ns = reference_host_ns + ns_per_cycle * (cycle - reference_cycle). Save maximum fit residual and round-trip time; they are evidence about alignment quality, not decoration.

Use several samplesTwo points define a line but cannot reveal jitter. Collect samples before and after the workload, reject malformed/outlier exchanges by an explicit policy, and retain the raw samples.
Fit each domain independentlyController, array sequencer, converter, DSP, and a second chip may have different clocks. Never order unrelated raw cycle values directly.
Start a new epochReset, clock gating, DVFS, wrap, suspend/resume, firmware restart, or device replacement invalidates the old map unless the hardware contract proves continuity.
Expose uncertaintyRound-trip asymmetry and fit residual bound confidence in cross-domain ordering. Events closer than the uncertainty may be concurrent; do not invent a precise order.

Code styleFormat automatically; compile with strong warnings; use static analysis; keep public headers minimal; prefer typed units, enums and spans over unstructured integers and pointers.
Error behaviorStable status category, subsystem, operation/graph ID, device ID, cause, recovery hint and machine-readable fields. Do not make users decode a generic timeout.
LoggingStructured events with severity and correlation IDs. Avoid hot-loop logging and sensitive/raw tensor dumps by default.
Commit qualityDescriptive subject, one coherent purpose, builds independently, relevant tests pass, generated output and docs remain synchronized.
Review depthArchitecture/schema changes need contract owners; kernel performance changes need correctness and profiler evidence; model claims need reproducible manifests.
Definition of doneImplementation, focused regression test, required lanes, observable failure path, documentation, compatibility decision, and measured acceptance evidence.

Debugging playbook

Find the first layer that diverges, then instrument only that layer

TT-Metal treats debug print, Watcher, Inspector, Tracy, device profiling, assertions, triage and NoC dumps as explicit tools. An ACiM repository needs equivalent host, compiler, runtime, device, numerical and performance views.

1. ReproduceFreeze the runCommit, target, artifact, calibration, seed, inputs, backend
2. ClassifyName the symptomBuild, compile, load, hang, wrong result, intermittent, performance
3. CompareFind first divergenceReference, ideal array, nonideal model, queued sim, RTL, silicon
4. InspectNarrow the layerIR, artifact, buffers, queue, firmware, array, digital epilogue
5. ProveMinimize and fixSmall vector, deterministic failure, explicit invariant
6. PreserveAdd regressionFocused test, failure bundle, docs and applicable deeper lane
Host C++Memory, lifetime, threadsDebug symbols, GDB/LLDB, ASan, UBSan, TSan, core dumps, queue/event state and ownership logs.Start: failing GoogleTest under sanitizer
Python and bindingsAPI, conversion, exceptionspytest -x -vv, one node ID, faulthandler, typed boundary logs, native debugger for binding crashes.Start: installed package, one parametrization
CompilerSemantics and loweringVerify after every pass, dump canonical IR, pass bisect, target legality report, mapping explanation, deterministic artifact diff.Start: first different IR operation
Artifact and runtimeCompatibility and executionArtifact inspector, schema/ABI versions, buffer ownership, command list, correlation IDs, queue timeline, timeout reason and cleanup trace.Start: load transcript and command ID
Firmware and deviceHang, fault, bad commandWatcher-style health monitor, waypoints, device print, bounded asserts, command validation, register snapshot, interrupt history, safe reset and triage bundle.Start: last completed waypoint
Analog numerical pathAccuracy and variabilityLayerwise reference, ideal-array, characterized nonideality and silicon outputs; converter saturation, slice sums, calibration, remap and defect-map identity.Start: first tensor exceeding tolerance
PerformanceLatency and utilizationSame artifact and machine state, warmup, host/device timeline, DMA bytes, queue waits, array active time, ADC/DSP occupancy, stalls and thermal history.Start: phase with changed cycles or bytes
Intermittent failuresRace, thermal, state leakRepeat with fixed seed, record preceding tests, serialize device use, vary reset depth, track temperature, allocator state, calibration age and firmware logs.Start: smallest repetition count that fails
Scale-outRoute and collective failureTopology epoch, logical/physical map, route table, per-link counters, collective step, timeout ownership and before/after recovery trace.Start: first node missing progress

Do not enable every device tool at once. Instrumentation consumes memory, bandwidth, timing margin, or device resources and can change a race or performance result. TT-Metal documentation explicitly notes conflicts among some device debug/profiling modes; document equivalent exclusions in the new stack.

Developer loop

Give contributors short, memorable commands for every test depth

These are proposed command names for the blueprint, not commands currently supplied by TT-Metal.

Local build and focused verificationIllustrative interface to implement in scripts and presets
# Configure, build and install a host-only development tree
cmake --preset dev
cmake --build --preset dev --parallel
cmake --install build/dev --prefix build/install

# Fast checks before a pull request
pre-commit run --all-files
ctest --preset host-unit --output-on-failure
python -m pytest tests/python/unit -m "unit and not hardware"

# One complete simulator path
python -m pytest tests/smoke/simulator -m smoke
acim-compile models/manifests/matmul.yaml -o build/matmul.acx
acim-run build/matmul.acx --backend functional --trace build/matmul.atrace

# Reserved hardware path
python -m pytest tests/smoke/hardware -m "hardware and smoke" --device auto

# Broader correctness and performance lanes
python -m pytest tests/sweeps -m nightly --maxfail=20
python -m pytest tests/end_to_end -m "model_tier_1"
python -m pytest tests/performance -m performance --benchmark-manifest configs/performance.yaml

# Diagnostic host/device profile; compare against profile-off benchmark
cmake --preset tracy
cmake --build --preset tracy
ctest --preset tracy

# Reproduce a failure bundle
acim-triage collect --run-id RUN_ID --output build/triage/RUN_ID
acim-triage compare --left functional --right hardware --first-divergence

A real implementation should wrap lab reservation, timeout, artifact capture and reset in fixtures or runner services so individual tests cannot leave a shared device unusable.
