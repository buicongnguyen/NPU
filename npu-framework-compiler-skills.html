Deep-learning framework internals

Backend engineers must know which semantics survive capture

Eager execution, automatic differentiation, graph capture, ahead-of-time export, and backend lowering are different mechanisms. A graph compiler receives only the behavior that the selected capture path represents.

Dynamic execution

PyTorch eager: Python call to dispatched kernel

A tensor operation reaches the dispatcher, which selects behavior for backend and cross-cutting features such as autograd, autocast, or functionalization. Autograd records a reverse-mode graph of Function objects as the forward computation executes.

Reason aboutDispatch keys, ATen operator schema, tensor metadata, views/aliasing, mutation, saved tensors, grad mode.
Backend implicationAn NPU kernel registration is not enough if autograd, fake/meta, autocast, serialization, or profiling paths are missing.
Failure signatureWrong dispatch key, missing kernel, device mismatch, alias violation, unsupported layout, or backward error.
EvidenceOperator registration tests, gradcheck where applicable, fake-tensor shape tests, profiler events, CPU differential vectors.
Captured execution

PyTorch compile and export are related but not interchangeable

torch.compile uses TorchDynamo to capture Python frames, AOTAutograd to capture forward/backward work, and a backend such as TorchInductor to generate code. Guards and graph breaks preserve dynamic Python behavior. torch.export targets an ahead-of-time, full-graph ExportedProgram with explicit input constraints.

DynamoCaptures traceable regions and guards assumptions; unsupported behavior can create graph breaks or recompilation.
AOTAutogradFunctionalizes and captures forward/backward graphs so compilers can optimize beyond eager autograd execution.
Inductor / backendConsumes captured graphs, performs fusion/scheduling, and generates target code or delegates to another backend.
ExportRepresents an AOT program plus constraints; deployment must validate shapes and unsupported Python side effects.
Trace and cache

TensorFlow: eager code becomes one or more concrete graphs

tf.function traces TensorFlow operations into graphs and caches concrete functions for compatible inputs. AutoGraph converts supported Python control flow into graph-building code. Python side effects happen during tracing, not as ordinary work on every graph execution.

Reason aboutTrace keys, TensorSpec signatures, polymorphic shapes, retracing, captured variables, resources, control flow.
Optimization pathGraph optimizers and optional XLA compilation operate after capture; target clusters must preserve resource and control dependencies.
Failure signatureExcess retracing, Python value frozen at trace time, unsupported op, resource placement, shape specialization, graph/eager mismatch.
EvidenceConcrete-function inventory, GraphDef inspection, signature tests, eager/graph parity, target cluster and fallback report.

ONNX network analysis

Inspect the graph before optimizing it

ONNX separately versions the IR, operator sets, and model. Shape inference is useful but intentionally incomplete for some dynamic behavior. A production analyzer records unknowns instead of inventing shapes.

1ValidateIR, opsets, schemas, topology, external data
2InferTypes, ranks, static and symbolic dimensions
3InventoryOps, attributes, constants, subgraphs, state
4DiagnoseTarget predicates, dynamic regions, fallback islands
5TransformOrdered passes with local proof obligations
6VerifyStructural checks, numerical parity, performance delta
ONNX forensic inventoryPython
from collections import Counter
import math
import numpy as np
import onnx
from onnx import checker, helper, shape_inference

model = onnx.load("model.onnx", load_external_data=True)
checker.check_model(model, full_check=True)
typed = shape_inference.infer_shapes(model, strict_mode=False)
graph = typed.graph

def shape_of(value):
    tensor = value.type.tensor_type
    if not tensor.HasField("shape"):
        return "unknown-rank"
    dims = []
    for dim in tensor.shape.dim:
        dims.append(dim.dim_value or dim.dim_param or "?")
    return dims

print("IR:", typed.ir_version)
print("Opsets:", [(item.domain or "ai.onnx", item.version)
                  for item in typed.opset_import])
print("Operators:", Counter(node.op_type for node in graph.node))
print("Inputs:", [(value.name, shape_of(value)) for value in graph.input])
print("Outputs:", [(value.name, shape_of(value)) for value in graph.output])

def logical_bytes(tensor):
    if tensor.data_type == onnx.TensorProto.STRING:
        return sum(len(value) for value in tensor.string_data)
    dtype = np.dtype(helper.tensor_dtype_to_np_dtype(tensor.data_type))
    return math.prod(tensor.dims) * dtype.itemsize

weight_bytes = sum(logical_bytes(item) for item in graph.initializer)
print("Initializers:", len(graph.initializer), weight_bytes, "logical bytes")

for index, node in enumerate(graph.node):
    domain = node.domain or "ai.onnx"
    print(index, domain, node.op_type, list(node.input), list(node.output))

Analyzer outputs worth keeping in CI

  • IR and opset versions, model and external-data digests, producer metadata.
  • Input/output signatures with symbolic dimensions and bounded-shape policy.
  • Operator, domain, attribute, initializer, sparse tensor, function, and subgraph inventory.
  • Target support result for every node, with reason and estimated boundary transfer cost.
  • Quantization representation: QDQ, integer operators, scales, zero points, granularity, axis.
  • Reference vectors and per-output tolerances tied to preprocessing and postprocessing versions.
  • Peak live tensor estimate, constant bytes, dynamic allocation sites, and fallback memory.
Pass orderTransformationProof obligation before the next stage
1. Validate/versionReject malformed graphs; convert opsets only with defined adapters.Every node resolves to known semantics for its imported domain and version.
2. Type/shape analysisPropagate known facts and record unresolved symbols.No pass assumes a static dimension that analysis has not proved or guarded.
3. NormalizeCanonicalize equivalent forms and make implicit broadcasts/layouts explicit where needed.Reference output remains equal; source and quantization metadata survive.
4. SimplifyConstant fold, eliminate dead/identity work, simplify control and shape subgraphs.Folding respects dtype, overflow, NaN, rounding, and external-data policy.
5. Capability/partitionAssign maximal legal regions to NPU or fallback providers.Each region satisfies target predicates; boundary conversions are explicit and costed.
6. Target transformsFuse, choose layouts, quantize/lower, schedule, and plan memory inside owned regions.Numerical, memory, and resource constraints pass independent verifiers.
7. Package/compareEmit executable and run framework/ONNX/target parity plus performance measurements.Artifact provenance, tolerances, target versions, and regression result are retained.

Partition timing matters. ONNX Runtime performs provider-independent optimizations before partitioning and provider-specific work afterward. Its current documentation recommends a plugin execution provider for new integrations starting with ONNX Runtime 1.23. A vendor backend should verify which layer owns each rewrite so two optimizers do not erase one another's patterns or apply incompatible layouts.

RISC-V and NPU accelerator ISA

The key decision is how software submits work, not whether an opcode looks elegant

RISC-V reserves custom encoding spaces for vendor-specific extensions. An accelerator may instead use MMIO, memory-resident command queues, a standard device interface, or a combination. Workload granularity and system isolation usually matter more than instruction count.

Integration styleBest fitMain costCompiler/runtime consequence
Custom scalar/vector instructionFine-grain tightly coupled operations with registers or local state.Toolchain, context state, exceptions, virtualization, and memory-model integration.Intrinsic/builtin lowering, assembler/disassembler, ABI and scheduler awareness.
MMIO controlSimple control/status, boot, reset, capability, and low-rate commands.High per-command CPU overhead; ordering and posted writes require care.Driver API and register generation; rarely suitable for every tensor tile.
Memory command queueCoarse asynchronous kernels, DMA descriptors, multiple in-flight jobs.Descriptor validation, coherency, fences, queue wrap, reset and fault recovery.Compiler emits command graph; runtime owns buffers, dependencies, submission, events.
Firmware RPCComplex scheduling, stable host API across changing hardware internals.Firmware latency, version negotiation, watchdogs, diagnosability.Artifact carries firmware-facing operations; host remains insulated from microarchitecture.

struct npu_command_v1 {
    uint16_t size_bytes;       /* permits compatible extension */
    uint16_t opcode;
    uint32_t flags;
    uint64_t command_id;
    uint64_t executable_id;
    uint64_t input_iova;
    uint64_t output_iova;
    uint32_t input_bytes;
    uint32_t output_bytes;
    uint32_t dependency_count;
    uint32_t completion_slot;
};

/* Producer protocol:
 * 1. Validate ownership and bounds.
 * 2. Write descriptor and dependency records.
 * 3. Publish with the required release ordering.
 * 4. Update producer index and ring the doorbell.
 */

ISA and ABI review checklist

  • Define architectural state, privilege, context-switch cost, and reset value.
  • Specify operand widths, alignment, endianness, rounding, saturation, and exception behavior.
  • Separate virtual, physical, and I/O virtual addresses; state IOMMU and coherency assumptions.
  • Name memory ordering for descriptors, MMIO, DMA, completion, and CPU visibility.
  • Define malformed command, page fault, watchdog, engine fault, retry, and partial-write behavior.
  • Reserve feature bits and size-versioned messages; test old/new driver and firmware combinations.
  • Generate encoders, decoders, docs, simulator tables, and disassembly from one source of truth.
  • Run differential ISA tests plus whole-command random sequences through model and RTL.

RISC-V uses a relaxed memory model and explicit fences for required ordering. A queue protocol must translate that architectural rule into concrete producer/consumer barriers and device coherency operations. "Write the descriptor, then ring the doorbell" is incomplete until visibility and ordering are specified for every agent.

Classify

Find the limiting resource

Separate host dispatch, queue wait, DMA, memory bandwidth, engine occupancy, synchronization, power-state transition, and thermal throttling before selecting a lever.

Hypothesize

Predict two metrics

State the expected primary benefit and at least one regression risk. For example, a lower frequency may improve power but increase leakage energy and miss a latency deadline.

Validate

Use distributions and controls

Pin model, inputs, software versions, operating point, temperature, and concurrent load; compare repeated latency and energy distributions with accuracy unchanged.

GUI programming for NPU tools

A debugger GUI should be a view over stable analysis data

Qt's model/view architecture separates data from presentation. Keep trace decoding and tensor analysis off the UI thread, publish immutable or versioned result models, and make every visual selection addressable by stable IDs.

Data architecture

One selection identity across panes

Selecting a graph node should highlight its fused region, commands, timeline slices, buffers, counters, and tensor diffs through stable IDs rather than text-name matching.

Responsiveness

Stream and aggregate outside the UI thread

Workers parse traces and tensors, cancellation stops stale work, and views request bounded windows. Million-event traces should not become a million GUI objects.

Testability

Test models before pixels

Unit-test sorting, filtering, time conversion, diff thresholds, selection propagation, and empty/error states. Add screenshot tests only after the underlying data contracts are stable.

Technical soft skills

Cross-functional work becomes visible in engineering artifacts

Interface writing

Turn assumptions into contracts

Write version, ownership, state, ordering, units, error, timeout, and compatibility rules. A short executable schema plus adversarial examples is stronger than a long ambiguous design note.

Evidence discipline

Separate observation, inference, and decision

Attach traces, counters, inputs, versions, and uncertainty to claims. State what the evidence rules out, what remains possible, and the next cheapest discriminating experiment.

Cross-team debugging

Build a shared reproduction boundary

Reduce the failure to an artifact, input, expected output, component matrix, and first divergent layer/command. Assign investigation by evidence, not organizational blame.

Design review

Expose tradeoffs early

Present workload, constraints, alternatives, cost model, failure modes, observability, rollout, and rollback. Record the decision and the evidence that would cause it to change.

Code review

Review behavior at boundaries

Look for silent fallback, unchecked dimensions, lifetime races, nondeterministic pass order, ABI drift, missing negative tests, and metric changes before style details.

Incident learning

Repair the detection system

A useful postmortem adds a contract, assertion, counter, replay test, compatibility gate, or deployment safeguard so the same class of failure becomes cheaper to detect.
