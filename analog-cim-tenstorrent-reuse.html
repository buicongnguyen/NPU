Reusing Tenstorrent for an Analog CIM NPU

A contract-first method for deciding what to learn, reuse, adapt, or replace. Tenstorrent is a valuable open-source reference, but its lower stack encodes a digital Tensix machine. An analog compute-in-memory product needs its own array mapping, numerical contract, calibration state, weight lifecycle, and device backend.

Executive verdict

Reuse the boundary patterns first, not the hardware assumptions

The important correction

"Reuse Tenstorrent" is not one decision. Directly importing a model corpus is different from adapting an MLIR frontend, and both are very different from forking a runtime or kernel driver. Decide at component boundaries, and require semantic fit, legal clarity, tests, and an exit strategy for every component.

Mode 1

Learn

Read the code and documentation to understand compiler layering, explicit data movement, tests, tooling, and release practices. No code dependency is created.

Mode 2

Reuse

Adopt target-neutral code with a compatible license and a stable boundary. Keep upstream history, attribution, tests, and dependency records.

Mode 3

Adapt

Modify an isolated component whose concepts fit but whose target interface does not. Budget for rebasing, qualification, and divergence.

Mode 4

Replace

Implement a new analog backend when the original code assumes Tensix cores, tiles, circular buffers, NoC transactions, or Tenstorrent device ABIs.

2026 repository audit: the strongest additions are Tenstorrent's reusable SoC/cluster descriptor boundary, MeshDevice virtual-device model, static cluster descriptors, tt-emule alternate-backend pattern, TT-Fabric control-plane design, and report tooling. The scale-out, fault-tolerance, and LLM design turns those patterns into concrete ACiM interfaces without importing Tensix execution semantics.

Hardware disclosure gate

No backend decision is credible until this contract is known

IHW's public patent record reveals part of the numerical architecture: positional digit-sliced weights, adaptive converter range, grouped quantization, and signed-current cancellation. It does not show that every pending mechanism is implemented in a product, or disclose the complete array, command, calibration, compiler, firmware, and runtime contract needed for a compatibility judgment. Treat the following as interview and onboarding questions, not assumptions about its shipping design.

New evidence: the iHW patent deep read separates granted claims, pending applications, company-reported silicon, remaining accuracy errors, and the metadata an ACiM compiler and runtime would need.

Architecture mismatch explorer

A digital dataflow processor and an analog array expose different contracts

Different primitive

Core-grid kernels versus physical array mappings

TenstorrentPrograms reader, compute, and writer kernels across Tensix cores using tiles, circular buffers, and NoC traffic.
Analog CIMPrograms conductance cells, encodes inputs, performs array MVM, digitizes columns, and combines slices or partial sums.
Compiler consequenceReplace core placement with physical array tiling, signed mapping, converter assignment, and analog/digital partitioning.
ProofOne matrix must map to arrays and produce an explainable artifact with no hidden Tensix assumptions.
Different error contract

Quantization is only one part of analog numerical behavior

Digital pathDeclared formats, rounding, saturation, and deterministic instruction semantics dominate.
Analog pathConverter quantization, programming error, read noise, nonlinear response, wire effects, temperature, and drift interact.
Compiler consequenceCarry range, slice, ADC, calibration, and tolerance metadata through lowering and preserve it in the artifact.
ProofLayerwise error distributions and task accuracy must correlate across reference, simulator, model, and silicon.
New persistent state

The executable depends on the physical device population

Digital pathWeights are ordinary bytes loaded into memory; two compliant devices should execute the same bits.
Analog pathProgrammed conductances, calibration coefficients, fault maps, drift age, and thermal state can differ by chip.
Runtime consequenceVersion and bind model artifacts to device-specific state without making builds irreproducible.
ProofCold boot, recalibration, field update, rollback, replacement device, and corrupted-state recovery all pass.
Different cost model

Programming and conversion join compute and data movement

Tenstorrent focusTile locality, SRAM capacity, NoC bandwidth, kernel overlap, and core utilization.
Analog additionsArray residency, programming amortization, ADC contention, slice count, calibration, remapping, and digital fallback.
Scheduler consequenceOptimize end-to-end latency and energy under accuracy, endurance, temperature, and memory constraints.
ProofCost predictions explain measured bottlenecks and remain stable across representative shapes and corners.
Wider qualification

Correctness is a population and lifecycle claim

Digital qualificationInstruction correctness, timing, memory, concurrency, firmware compatibility, power, and reliability.
Analog additionsDevice distributions, yield, retention, endurance, drift, recalibration overhead, thermal corners, and aging.
Operations consequenceTelemetry and health policy need thresholds, fallback, quarantine, reset, and service decisions.
ProofPopulation statistics support the product SLA rather than one favorable chip or one benchmark run.

Recommended architecture

Keep the portable graph stack above an explicit ACiM boundary

Tenstorrent can remain a comparison target or reference implementation without becoming the core dependency of the analog backend.

Design rule: the target-neutral IR should describe model meaning, while ACiM IR should describe physical analog intent. Neither layer should pretend that a conductance array is a Tensix core.

Analog-specific work

The custom backend begins where physical state changes correctness

Implementation companion: this page decides what to learn, adapt, or replace from Tenstorrent. The SDK and compiler guide turns that decision into concrete libraries, APIs, schemas, compiler passes, simulator backends, tests, and a first runnable slice.

Mapping

Array tiling and decomposition

Split matrices by physical rows, columns, differential representation, bit slices, ADC lanes, and available arrays. Define partial-sum combination and padding.

Evidence: mapping report includes utilization, slices, conversions, predicted error, and digital work.

Numerics

Range and converter planning

Select scales, input encoding, accumulation range, ADC mode, saturation policy, and per-layer tolerance using calibration and workload distributions.

Evidence: reference versus simulated histograms identify the first divergence and its physical cause.

Partition

Analog and digital placement

Keep dense supported MVMs on arrays; assign nonlinear, control, normalization, unsupported, or precision-sensitive work to digital engines or CPU fallback.

Evidence: capability report states every fallback and conversion cost.

Programming

Conductance and fault-aware placement

Translate logical weights into programming targets, write/verify actions, remapped cells, spare resources, and per-chip state with endurance constraints.

Evidence: deterministic programming manifest and post-program verification report.

Calibration

Compensation insertion

Attach calibration versions, converter corrections, array gains/offsets, drift models, temperature ranges, and recalibration triggers to executable regions.

Evidence: stale or incompatible calibration is rejected or repaired explicitly.

Artifact

Device-aware packaging

Separate portable model content from device-bound programming and calibration state so builds, deployment, updates, rollback, and replacement remain auditable.

Evidence: schema validation and ABI compatibility tests cover every section.

Open-source engineering

A repository label is not a complete license decision

This is an engineering checklist, not legal advice. Employer legal counsel or the open-source program office should approve product use and distribution.

Inventory

Record provenance per component

Capture repository, commit, file/module boundary, SPDX identifier, copyright, modifications, build options, generated code, models, datasets, and transitive dependencies.

Obligations

Preserve notices and license texts

Apache-2.0 code can carry NOTICE, attribution, patent, and trademark considerations. Dependencies may use different licenses even when the top-level project is Apache-2.0.

Boundary

Review GPL components separately

TT-KMD is GPL-2.0 while TT-Metal, TT-MLIR, and TT-UMD publish Apache-2.0 licenses. Architecture and distribution choices must not collapse those distinctions.

Maintenance

Choose upstream, vendor, or fork deliberately

Define patch ownership, security response, upgrade cadence, compatibility tests, contribution policy, and the cost of carrying target-specific divergence.

Company due diligence

Interview questions that reveal the real engineering environment

Architecture

What is already fixed?

  • What is the analog/digital execution boundary?
  • Which array, converter, programming, and calibration parameters are compiler visible?
  • Which first models and accuracy/latency/energy SLAs define success?
  • Is a versioned architecture specification available to software engineers?
Software

What exists today?

  • Is the stack based on MLIR, LLVM, TVM, StableHLO/PJRT, or in-house infrastructure?
  • How mature are the simulator, compiler, runtime, firmware, driver, profiler, and layer debugger?
  • Who owns programming, calibration, per-chip state, and field recovery?
  • How are model, simulator, RTL, and silicon results correlated?
Team

What would I own?

  • Is the expectation architecture, implementation, bring-up, or all three?
  • What should be working after the first 30, 60, and 90 days?
  • What is the open-source intake and upstream-contribution policy?
  • Which proprietary data may approved coding assistants access?

Signals that require a deeper conversation

No versioned hardware contract; no golden numerical reference; accuracy claims without chip-population data; compiler passes compensating for undocumented hardware behavior; calibration owned by nobody; a proposed fork without maintainers; or "open source" treated as a substitute for license review.

Coding-assistant policy

Use assistants for implementation without outsourcing architecture or trust

Information

Use only approved environments

Do not submit proprietary RTL, process data, calibration records, customer models, security material, or unreleased specifications to an unapproved service.

Provenance

Track generated and imported code

Review origin, license, dependencies, generated-file headers, security findings, and repository history. Never accept an assistant's license claim without source verification.

Verification

Require ordinary engineering evidence

Generated code needs human ownership, unit and differential tests, static analysis, hardware-model correlation, performance measurement, and normal code review.

Judgment

Keep decisions with the engineering team

Assistants can explain code and draft implementations. Hardware contracts, numerical tolerances, safety policy, product tradeoffs, and sign-off remain accountable human decisions.
