Architecture mismatch explorer

A digital dataflow processor and an analog array expose different contracts

Different primitive

Core-grid kernels versus physical array mappings

TenstorrentPrograms reader, compute, and writer kernels across Tensix cores using tiles, circular buffers, and NoC traffic.
Analog CIMPrograms conductance cells, encodes inputs, performs array MVM, digitizes columns, and combines slices or partial sums.
Compiler consequenceReplace core placement with physical array tiling, signed mapping, converter assignment, and analog/digital partitioning.
ProofOne matrix must map to arrays and produce an explainable artifact with no hidden Tensix assumptions.
Different error contract

Quantization is only one part of analog numerical behavior

Digital pathDeclared formats, rounding, saturation, and deterministic instruction semantics dominate.
Analog pathConverter quantization, programming error, read noise, nonlinear response, wire effects, temperature, and drift interact.
Compiler consequenceCarry range, slice, ADC, calibration, and tolerance metadata through lowering and preserve it in the artifact.
ProofLayerwise error distributions and task accuracy must correlate across reference, simulator, model, and silicon.
New persistent state

The executable depends on the physical device population

Digital pathWeights are ordinary bytes loaded into memory; two compliant devices should execute the same bits.
Analog pathProgrammed conductances, calibration coefficients, fault maps, drift age, and thermal state can differ by chip.
Runtime consequenceVersion and bind model artifacts to device-specific state without making builds irreproducible.
ProofCold boot, recalibration, field update, rollback, replacement device, and corrupted-state recovery all pass.
Different cost model

Programming and conversion join compute and data movement

Tenstorrent focusTile locality, SRAM capacity, NoC bandwidth, kernel overlap, and core utilization.
Analog additionsArray residency, programming amortization, ADC contention, slice count, calibration, remapping, and digital fallback.
Scheduler consequenceOptimize end-to-end latency and energy under accuracy, endurance, temperature, and memory constraints.
ProofCost predictions explain measured bottlenecks and remain stable across representative shapes and corners.
Wider qualification

Correctness is a population and lifecycle claim

Digital qualificationInstruction correctness, timing, memory, concurrency, firmware compatibility, power, and reliability.
Analog additionsDevice distributions, yield, retention, endurance, drift, recalibration overhead, thermal corners, and aging.
Operations consequenceTelemetry and health policy need thresholds, fallback, quarantine, reset, and service decisions.
ProofPopulation statistics support the product SLA rather than one favorable chip or one benchmark run.

TT-Metal performance patterns translated

Keep the dataflow machinery, but move it around resident analog weights

TT-Metal optimizes a digital machine whose operands move through DRAM, local SRAM, circular buffers, a NoC, and compute engines. ACiM removes most steady-state weight traffic when conductances remain programmed, but it still has commands, activations, converter traffic, partial sums, digital epilogues, and outputs to move. The useful abstraction is therefore not "no data movement"; it is "weight-stationary dataflow with explicit movement and synchronization around the array."

KeepDouble buffering

Ping-pong activation ingress and digitized-result egress so DMA and digital work can overlap array execution.

AdaptPrefetch

Prefetch commands and activations every run. Stage weights only before planned program/verify work; do not page conductances per token.

KeepFast dispatch

Let a device controller consume queued descriptors without waiting for one host call per layer or array firing.

ExtendProgram cache

Cache schedules and digital kernels, but bind every hit to the target, weight map, calibration, health, and ABI epochs.

CB: bounded producer-consumer storage CQ: ordered device work stream DMA: transfer engine without CPU copying NoC: on-chip packet network DAC: optional multi-level input converter ADC: sensed analog result to digital code MVM: array compute primitive

The bottleneck moves; it does not disappear

In steady state, an ACiM layer should not stream its full weight matrix from DRAM. The likely critical path is activation delivery, input encoding, row/word-line drive, optional DAC conversion, array settle time, ADC conversion, bit-slice reconstruction, digital reduction, and output movement. Once the pipeline is filled, its ideal tile interval is approximately max(Tin, Tfront_end + Tmvm + Tadc, Tepilogue, Tout), but only stages backed by independent hardware can overlap. Shared converters, power rails, thermal limits, or SRAM banks must be modeled as one serialized resource even when software exposes several queues.

Cache lifetime model

Reuse three different objects without confusing their validity

1. Kernel binary cache Compiler and ISA lifetime Stores DSP/vector kernels and firmware fragments. Key by source or IR, compiler flags/version, ISA, target revision, and firmware ABI. Invalidated by code generation or binary compatibility changes.
2. Program template cache Structural schedule lifetime Stores placement, CB plan, engine sequence, synchronization, and parameter slots. Mutable addresses and state handles remain runtime arguments. Invalidated by shape, layout, precision, topology, mapping, resource, or ABI changes.
3. Bound trace / residency cache Device-state lifetime Stores replayable descriptors bound to concrete buffers, programmed weights, calibration, health, and stable dependencies on one device. Invalidated by address movement, reprogram/remap, profile update, drift/health epoch, reset, or replacement device.

Runtime decision graph

Choose an optimization from observable hardware facts

1Are the required weights programmed and valid?Yes - use the resident handle; do not prefetch weights. No - enqueue program/verify or page replacement before execution.
2Can DMA overlap converter and array use?Yes - allocate ingress ping-pong buffers. No - keep the queue contract, but do not claim hidden latency.
3Do arrays share ADCs, input drivers/DACs, SRAM banks, routes, or power?Yes - model tokens for each shared resource and cap concurrency. No - schedule independent array groups in parallel.
4Are shape, addresses, weights, and profile epochs stable?Yes - replay a resident trace. No - use a cached parameterized program template or compile a new variant.
5Does one activation feed several resident weight shards?Yes - multicast once and synchronize receivers. No - use local or unicast delivery.

Analog-specific work

The custom backend begins where physical state changes correctness

Implementation companion: this page decides what to learn, adapt, or replace from Tenstorrent. The SDK and compiler guide turns that decision into concrete libraries, APIs, schemas, compiler passes, simulator backends, tests, and a first runnable slice.

Mapping

Array tiling and decomposition

Split matrices by physical rows, columns, differential representation, bit slices, ADC lanes, and available arrays. Define partial-sum combination and padding.

Evidence: mapping report includes utilization, slices, conversions, predicted error, and digital work.

Numerics

Range and converter planning

Select scales, input encoding, accumulation range, ADC mode, saturation policy, and per-layer tolerance using calibration and workload distributions.

Evidence: reference versus simulated histograms identify the first divergence and its physical cause.

Partition

Analog and digital placement

Keep dense supported MVMs on arrays; assign nonlinear, control, normalization, unsupported, or precision-sensitive work to digital engines or CPU fallback.

Evidence: capability report states every fallback and conversion cost.

Programming

Conductance and fault-aware placement

Translate logical weights into programming targets, write/verify actions, remapped cells, spare resources, and per-chip state with endurance constraints.

Evidence: deterministic programming manifest and post-program verification report.

Calibration

Compensation insertion

Attach calibration versions, converter corrections, array gains/offsets, drift models, temperature ranges, and recalibration triggers to executable regions.

Evidence: stale or incompatible calibration is rejected or repaired explicitly.

Artifact

Device-aware packaging

Separate portable model content from device-bound programming and calibration state so builds, deployment, updates, rollback, and replacement remain auditable.

Evidence: schema validation and ABI compatibility tests cover every section.

Decision 1 | 3 choices

Which model entry should you implement first?

Choose the boundary that matches the models and framework behavior you need to observe.

Recommended firstDirect ONNX importerBest for learning compiler semantics with YOLO and common deployment models. It keeps the first debug path small: model, importer, IR, lowering, simulator.
Integration stageONNX Runtime plugin EPBest when applications already use ORT and the compiler/runtime API is stable enough to support subgraph capability, compilation, caching, and CPU fallback.
OpenXLA pathStableHLO plus PJRTBest when JAX, XLA, or StableHLO portability is a product requirement. PJRT provides a uniform device API, not the ACiM lowering itself.

Recommended sequence: direct ONNX import for the first runnable slice, then an ORT plugin EP; add StableHLO/PJRT when OpenXLA clients are an explicit requirement.

Decision 2 | 3 choices

Which compiler foundation should carry ACiM semantics?

Choose based on how many abstraction levels, custom types, and target legality rules the compiler must express.

Recommended product pathMLIR dialect stackBest for a real ACiM backend with target-neutral tensors, analog-aware operations, physical mapping, command IR, verifiers, and progressive legalization.
Exploration pathTVM backendUseful for schedule search, tensor-expression experiments, and Python-centered prototyping when the analog state and artifact model remain modest.
Smallest experimentCustom C++ or Python IRUseful for one operator and quick cost-model experiments. It becomes expensive when adding graph regions, diagnostics, serialization, rewrites, and multiple targets.

Recommended for this architecture: MLIR, because ACiM needs several explicit levels: model meaning, analog intent, physical placement, and executable commands. Use a minimal Python model alongside it as the numerical oracle.

Decision 3 | 3 levels

Which execution backend should you build now?

These are successive fidelity levels rather than competitors; each reveals a different class of bug.

Start hereFunctional and nonideality modelTests operator semantics, mapping, quantization, converter range, variation, and layerwise accuracy without queue or timing complexity.
Build secondQueued or cycle-oriented fake deviceTests artifacts, buffers, dependencies, command ordering, timing estimates, trace IDs, timeouts, and recovery while hardware is unavailable.
Correlate thirdFPGA or silicon backendTests actual transport, firmware, converters, calibration, PVT behavior, faults, performance, and energy.

Recommended flow: implement all three behind one HAL in this order. Skipping the functional model mixes numerical bugs with runtime bugs; skipping the queued model makes hardware bring-up unnecessarily opaque.

Decision 4 | 4 choices

Where should each graph operation execute?

The compiler should choose per operation and shape; no single engine is best for the whole model.

Recommended policyHybrid cost-based partitionEvaluate accuracy, conversion, transfer, memory, latency, and energy for every candidate placement, then keep every fallback visible in the artifact and report.
Dense computeACiM arrayBest for large, supported, weight-resident MVM or convolution regions whose mapping, converter range, and measured accuracy satisfy the target.
Digital companionDSP or vector engineBest for activation, normalization, reduction, softmax, partial-sum repair, small tensors, and precision-sensitive work near the arrays.
Host fallbackCPUBest for rare unsupported operations, dynamic control, preprocessing, postprocessing, and early bring-up; transfer and synchronization costs must be visible.

Recommended rule: place an operation on ACiM only when end-to-end cost includes conversions, partial sums, calibration, and error recovery. Otherwise choose the closest digital engine, with CPU as the explicit final fallback.

Decision 5 | 3 maturity stages

How should host software reach the device?

Match the access path to the hardware stage and isolation requirements.

Start hereSimulator HALNo kernel driver is needed. The runtime calls an in-process or service backend implementing the same buffer, submit, wait, trace, and reset contracts.
Trusted labVFIO prototypeUseful for a single-purpose PCIe lab path with IOMMU-protected userspace access while the production UAPI is still changing.
Production deviceUMD plus KMDBest for multi-process isolation, DMA ownership, interrupts, reset, power, firmware, security, cleanup, and stable deployment. An integrated SoC uses equivalent platform-driver responsibilities.

Recommended flow: stabilize the HAL with simulator backends, use VFIO only for controlled bring-up when appropriate, then implement the production UMD/KMD boundary from the measured device contract.
