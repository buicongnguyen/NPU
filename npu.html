From TT-Metalium docs

Technical Deep Dive

TT-Metalium exposes more hardware detail than a typical framework runtime. That makes it valuable for NPU compiler study because the programmer sees where compiler decisions land: core placement, tile layout, SRAM buffering, NoC transfers, and kernel synchronization.

1. Tensix is a grid-first accelerator model

Tenstorrent describes the accelerator as a grid of specialized compute nodes called Tensix cores. Each core includes local L1 SRAM, two NoC interfaces, data movement control, a matrix engine/FPU, a vector engine/SFPU, pack/unpack hardware, and small RISC-V controllers that issue commands to those engines.

This is different from CUDA-style SIMT thinking. Instead of hiding latency behind many threads, the Metalium mental model is software-pipelined dataflow: explicitly move tiles, overlap movement with compute, and keep useful intermediate data close to the core.

2. The basic operation is reader -> compute -> writer

A typical TT-Metalium program creates three kernel roles. The reader data-movement kernel moves tiles from DRAM or another core into L1 circular buffers. The compute kernel waits on those buffers, runs tile operations through FPU/SFPU engines, and pushes output tiles to another circular buffer. The writer data-movement kernel drains output tiles and writes them back to DRAM or sends them onward.

The compiler lesson is direct: an NPU backend is not only selecting arithmetic instructions. It must schedule producer-consumer movement, buffer capacity, tile order, and synchronization so compute does not stall on memory.

3. Circular buffers are the local synchronization contract

Circular buffers are small FIFO queues in core-local SRAM. They are the connection point between data movement kernels and compute kernels. Calls such as cb_reserve_back, get_write_ptr, cb_push_back, cb_wait_front, and cb_pop_front express producer-consumer ownership.

In performance terms, the circular buffer depth controls how much movement and compute can overlap. Too little buffering starves compute. Too much buffering wastes scarce L1 SRAM that could hold other intermediate data.

4. Tiles and page size are core compiler concerns

Tenstorrent hardware is optimized around tiled tensors. A common tile shape is 32x32 elements. A tensor may be converted from row-major host layout into a tiled layout so the matrix/vector engines can consume data efficiently. Padding can be required when dimensions are not tile aligned.

This makes page_size, tensor layout, tilization, untilization, and sharding more than storage details. They become backend decisions that affect NoC traffic, L1 residency, and whether the compute engine receives data in the shape it wants.

5. NoC and memory placement replace hidden caches

TT-Metalium documentation emphasizes explicit data placement. The model is not "trust the cache." DRAM, local L1 SRAM, remote core SRAM, interleaved buffers, and sharded tensors are visible design choices. The NoC moves data between those locations.

For an NPU compiler engineer, this maps to locality decisions: keep reused blocks close to the consuming core, use multicast when many cores need the same data, shard tensors when the access pattern matches the core grid, and avoid unnecessary trips to DRAM.

6. Work distribution is explicit SPMD

TT-Metalium supports SPMD, but the developer or compiler decides which cores receive work and which runtime arguments each core sees. Utilities such as split_work_to_cores help divide tiles across available cores, but the program still needs correct per-core arguments.

This is exactly where compiler scheduling meets runtime setup: a graph partition becomes core ranges, tile ranges, reader/writer offsets, per-core work counts, and no-op behavior for unused cores.

Terminology

TT-Metalium Terms to Know

This glossary turns Tenstorrent documentation terms into NPU compiler study notes.

Tensix processor

The full accelerator chip or device organization built from a grid of nodes, including Tensix compute cores plus memory, PCIe, Ethernet, and management nodes.

Tensix core

The compute tile/node. It contains local L1 SRAM, NoC interfaces, RISC-V controllers, matrix/vector engines, and pack/unpack hardware.

L1 SRAM

Core-local scratchpad memory used for circular buffers, temporary tiles, and kernel code. It is explicit storage, not a transparent cache.

NoC

Network-on-Chip. The on-chip interconnect used to move data among DRAM, Tensix cores, and other device resources.

Tile

A hardware-friendly tensor block, commonly 32x32 elements, used by matrix and vector engines for efficient local computation.

Tilize / untilize

Conversion between row-major host-style layout and Tenstorrent tiled layout. This can happen on device to reduce host-side overhead.

Circular buffer (CB)

A small FIFO queue in L1 SRAM. Reader, compute, and writer kernels use CBs to exchange tiles safely and asynchronously.

Reader kernel

A data movement kernel that usually reads tiles from DRAM or another core into input circular buffers.

Compute kernel

The kernel that issues FPU/SFPU work, consumes input circular buffers, writes destination registers, and packs result tiles.

Writer kernel

A data movement kernel that waits for output circular buffers and writes result tiles to DRAM or another destination.

FPU / matrix engine

The matrix/tensor computation engine used for tile operations such as matmul and elementwise arithmetic.

SFPU / vector engine

The vector unit for scalar/vector style functions such as exp, log, sin, comparisons, and custom SFPI operations.

Pack / unpack

Hardware stages that move tiles between circular buffers, engine registers, and output buffers in the format the engines expect.

Runtime arguments

Per-kernel and per-core parameters such as buffer addresses, tile counts, work offsets, and core-specific ranges.

SPMD

Single Program, Multiple Data. Multiple cores run the same kernel program with different runtime arguments and tile ranges.

Multicast

A data movement pattern that sends the same data to multiple cores, useful for reuse in multi-core matmul and attention-like workloads.

Interleaved memory

A default memory placement style that spreads pages across memory controllers for balanced access.

Sharded tensor

A tensor distributed across multiple Tensix cores or memory regions to improve locality and reduce long-distance data movement.

Fast dispatch

An asynchronous command queue mode where the device processes queued work, reducing host CPU involvement compared with slow dispatch.

TT-NN

A higher-level Python tensor and operation library built on TT-Metalium. Useful for model-level experiments without writing every kernel manually.
