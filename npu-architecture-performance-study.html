The logic flow

Architecture starts with a scenario and ends with reproducible proof

Do not begin by choosing an array size or inventing opcodes. Each step consumes a reviewed artifact from the previous step and adds measurable obligations for the next one.

01 · CharacterizeModel workloadFamilies, operators, shapes, state, arrivals, precision, quality
02 · BudgetProduct targetsp50/p99 latency, throughput, energy, memory, cost, thermal
03 · ModelBounds and sweepsOperations, bytes, reuse, Roofline, sensitivity, bottleneck
04 · DesignArchitecture and ISAEngines, dataflow, hierarchy, commands, numerical rules
05 · RealizeCompiler and runtimeDSL, lowering, artifact, firmware, driver, execution, telemetry
06 · QualifyCorrelated product proofGolden, simulator, RTL, tape-out, silicon, SDK, sustaining

Keep three claims separate. Functional verification asks whether results and faults match the specification. Performance validation asks whether a fixed implementation meets its targets. Model correlation asks how closely one abstraction predicts another. Passing one does not imply the other two.

Architecture and ISA definition

Translate workload reuse and product constraints into explicit resources

A useful architecture document connects every resource to a workload need and every software-visible behavior to a test. The ISA is the stable contract; pipelines, arbitration, and implementation scheduling remain microarchitectural unless software can observe them.

Compute budgetrequired ops/s ≥ work ÷ deadlineInclude tail utilization, unsupported work, sparsity rules, and precision-specific peak rates.
Bandwidth budgetrequired B/s ≥ transferred bytes ÷ deadlineCount traffic at a named boundary: DRAM, NoC, SRAM bank, or register file.
Capacity budgetlive bytes = tensors + tiles + queues + stateInclude alignment, double buffers, fragmentation, firmware, and concurrent contexts.
Roofline boundrate ≤ min(peak, bandwidth × intensity)Operational intensity is work divided by bytes observed at the same memory boundary.

Normative contract

ISA and command definition checklist

  • Encoding, reserved values, decode precedence, capability discovery, and version negotiation.
  • Operand shapes, ranges, strides, layouts, alignment, address spaces, and alias rules.
  • Accumulation width, rounding, saturation, overflow, NaN behavior, determinism, and quantization.
  • Dependencies, barriers, fences, completion visibility, coherency, and memory ordering.
  • Privilege, isolation, IOMMU behavior, context state, cancellation, timeout, reset, and watchdog.
  • Illegal commands, page faults, engine faults, partial writes, recovery, and error propagation.
  • Debug state, counters, trace identity, assembler/disassembler, simulator, and conformance tests.
Example: one command as a testable contractIllustrative YAML
command: GEMM_V1
encoding:
  opcode: 0x18
  descriptor_bytes: 64
operands:
  A: {dtype: int8, layout: row_major, align: 64}
  B: {dtype: int8, layout: blocked_k32, align: 64}
  C: {dtype: int32, layout: row_major, align: 64}
semantics:
  value: C[m,n] = sum_k(A[m,k] * B[k,n])
  accumulator: signed_int32
  overflow: wrap_twos_complement
ordering:
  reads_after: dependency_tokens
  completion:
    order: release
    scope: system
    covers: [C_writes, status_record]
faults:
  bad_alignment: INVALID_DESCRIPTOR
  unmapped_iova: PAGE_FAULT
  unsupported_shape: UNSUPPORTED
observability:
  counters: [active_cycles, macs, sram_stall_cycles]
  trace: [command_id, start_tick, end_tick, status]

Convention warning: this page uses two arithmetic operations per multiply-accumulate when presenting GEMM/Conv operation counts. Hardware data sheets sometimes count one MAC as one operation and sometimes as two. State the convention before comparing throughput claims.

Modern model architectures and low-precision inference

Treat each model as a workload signature and each datatype as a full-stack contract

Model names change faster than silicon. Study the stable architectural signatures: dominant operators, tensor shapes, arithmetic intensity, persistent state, dynamic control, communication, and the end-to-end service goal. Then decide precision with measured task quality and real kernels—not bit width alone.

Normative contract

Write precision semantics before selecting peak TOPS

  • Encoding, exponent/mantissa or integer range, scale and zero-point ownership, and metadata layout.
  • Per-tensor/channel/group/block granularity, block shape, packing order, alignment, and tail rules.
  • Rounding mode, saturation or wrap, accumulator width, bias precision, reduction order, and determinism.
  • NaN, Inf, signed zero, subnormal, divide-by-zero, overflow, underflow, and conversion behavior.
  • Supported operator/shape/layout combinations, higher-precision islands, fallback, capability discovery, and revisioning.
Quality + systems proof

Avoid the low-bit performance illusion

  • Count scale, zero-point, packed-weight, padding, conversion, and temporary-buffer traffic.
  • Confirm that the target shape uses a native kernel rather than unpacking into a wider fallback.
  • Measure end-task perplexity, accuracy, generation or perceptual quality and worst cases—not layer MSE alone.
  • Separate theoretical footprint from allocated memory and theoretical operations/s from full-request latency.
  • Preserve one bit-accurate executable reference shared by compiler, simulator, RTL, firmware, and validation.

Conv, GEMM, and Attention

Use the same operator loop: semantics → work → traffic → mapping → proof

Operator names hide the cases that determine a product: depthwise and grouped Conv, skinny and batched GEMM, attention prefill versus decode, odd tails, dynamic shapes, masks, layouts, and low-precision edge behavior.

NCHW notation

Convolution is a family of reuse patterns, not one kernel

For output N×K×P×Q, filter K×(C/G)×R×S, and G groups, the conventional count is 2×N×P×Q×K×(C/G)×R×S operations. First derive P and Q from padding, stride, and dilation; then inspect reuse and tail behavior.

ClassifyRegular, 1×1, depthwise, grouped, dilated, or transposed; each changes reuse, reduction width, and array utilization.
MapChoose direct, implicit GEMM, explicit im2col, or Winograd; tile N/P/Q/K/C and account for halo fetches.
OptimizeRetain activations/weights/partial sums, align channels, double-buffer, compress zeros, and fuse bias/activation/pooling.
ProveRandomized shape parity, padding/dilation/groups, quantized rounding, edge tiles, traffic counters, and end-to-end layout cost.

Key trap: explicit im2col simplifies GEMM reuse but can replicate activations and inflate bandwidth. Implicit GEMM forms needed tiles during load, trading address-generation complexity for lower external traffic.

C[M,N] = A[M,K] × B[K,N]

GEMM performance follows the entire blocking hierarchy

The conventional count is 2×M×N×K operations. An idealized compulsory-traffic intensity is 2MNK ÷ (sA×MK + sB×KN + sC×MN), where sA, sB, and sC are bytes per element; add another C read when β is nonzero. Real intensity must also include packing, partial-sum spills, refetches, and the memory level being analyzed.

ClassifySquare, tall-skinny, short-wide, batched, transposed, split-K, and small-launch cases stress different resources.
MapBlock DRAM→SRAM→array/register levels; assign M/N spatially, reduce K, and select stationary operands.
OptimizePack and align vectors, pipeline DMA, double-buffer, fuse epilogues, balance split-K reduction, and specialize edge tiles.
ProveCompare reference output, achieved operations/s, array occupancy, per-level bytes, bank conflicts, launch overhead, and tails.

Key trap: a tile that maximizes MAC occupancy may exceed SRAM capacity, create bank conflicts, or force partial-sum spills. Optimize measured end-to-end time, not one utilization counter.

softmax(QKᵀ / √d + mask)V

Prefill is compute-rich; autoregressive decode is often KV-traffic dominated

Dense attention score work scales as O(Lq×Lk); it is quadratic when query and key lengths are the same. A naïve implementation also materializes the score matrix. IO-aware tiling with online softmax keeps score blocks on chip and is mathematically exact, although floating-point reduction reordering can still produce tolerance-level differences.

ClassifyPrefill versus decode, causal versus non-causal, self versus cross attention, MHA/GQA/MQA, batch, sequence, and head dimension.
MapTile Q/K/V, pipeline QKᵀ→online softmax→PV, retain running max/sum, and plan KV-cache layout and paging.
OptimizeFuse masking/scaling/softmax, overlap KV fetch, use grouped query where allowed, fuse dequantization, and avoid score materialization.
ProveStable softmax, fully masked rows, causal boundaries, variable lengths, cache update/reuse, long sequences, and low-precision error.

Key trap: changing the head count, attention approximation, or KV representation can change model semantics or quality. Separate an exact implementation optimization from a model-level approximation.

Frontend compilers, backend interfaces, accelerator runtimes, and domain-specific languages

Choose the smallest durable extension seam, then preserve semantics to the device

The frontend owns a verified IR handoff contract for the backend: graph optimization, operator lowering, dataflow and memory scheduling, IR invariants, and provenance. The backend turns that contract into target code; the runtime owns concrete execution and resource lifetimes. Correlated evidence must flow back across all three.

Internal customization policy

Make the fork burden visible

  • Write an extension-seam decision record covering ownership, fallback, performance, portability, integration cost, and expected lifetime.
  • Pin the upstream commit and toolchain; make builds reproducible; record patches, licenses, security response, and supported host platforms.
  • Isolate vendor dialects, targets, runtime modules, and feature flags. Keep unavoidable core patches small and review whether generic fixes can go upstream.
  • Budget rebases; test both the pinned baseline and candidate upgrades; retain IR reproducers for pass crashes and changed optimization decisions.
  • Qualify old and new model artifacts, compiler/runtime/driver versions, devices, and cached executables before an internal release.
Acceptance suite

Test decisions, not only generated answers

  • Golden import and IR snapshots, round trips, verifier negatives, pass determinism, and source-location preservation.
  • Differential operator/model results across boundary shapes, layouts, dtypes, quantization, dynamic dimensions, and hardware revisions.
  • Visible partition/fallback reasons, cost-model decisions, host/device copies, cache keys, compile time, and unsupported-input diagnostics.
  • Artifact load/reject behavior, buffer lifetime, stream ordering, cancellation, asynchronous errors, faults, and recovery.
  • Stable profiling IDs connecting framework node → IR operation → command → counter/trace event → device result.

Boundary diagnosis rule: preserve stable identity from framework node → graph IR → lowered operation → backend kernel/command → runtime event → device counter. First label a case unsupported, legal but not selected, or selected and failed. Then split latency into compile, load, allocation, copy, queue, device, and synchronization time; inspect reformats, spills, fragmentation, launches, fallback, and utilization. Change one factor, recheck correctness or task quality plus p50/p99, throughput, and peak memory, then feed the result into capability, cost, and architecture models.

Example accelerator DSL contractIllustrative, not executable syntax
kernel qgemm(A: i8[M,K], B: i8[K,N],
             a_scale: f32, b_scale: f32[N],
             c_scale: f32, C: i8[M,N])
  requires K % 32 == 0 and K <= 65535
    and a_scale > 0 and all(b_scale > 0) and c_scale > 0
    and aligned(A, 64) and aligned(B, 64)

  numerics {
    zero_points: A = 0, B = 0, C = 0
    accumulator: acc[m,n] = sum_k(i32(A[m,k]) * i32(B[k,n]))
    real_scale:  r[n] = a_scale * b_scale[n] / c_scale
    output:      C[m,n] = saturate_i8(round_nearest_even(acc[m,n] * r[n]))
  }

  schedule non_semantic {
    tile M by 64, N by 64, K by 32
    double_buffer A, B in scratchpad
    map [M, N] to matrix_engine
  }

  lowers_to @npu.gemm_v1
  fallback  @reference.qgemm
DSL design review

Specify meaning before syntax

  • Purpose: name the user and abstraction boundary. Separate algorithmic semantics from a retargetable schedule.
  • Types: define shapes, dynamic dimensions, layouts, precision/scales, memory spaces, ownership, aliases, effects, and synchronization.
  • Numerics: define accumulation, reduction order, overflow, saturation, rounding, NaNs, determinism, and approximation tolerance.
  • Legality: implement verifiers with source-located diagnostics and invalid-program tests; do not defer every error to code generation.
  • Lowering: retain before/after IR for semantic NPU → tiled accelerator → buffer/command → LLVM IR or custom ISA stages.
  • Evolution: version operations and artifacts, preserve profiling identity, provide a bounded escape hatch, and document deprecation.

Capstone proof: compile one fused graph through StableHLO or Relax → graph optimization → semantic NPU DSL → scheduled/bufferized IR → backend target code → executable + metadata → runtime → mock driver/device. Retain partition and lowering decisions, every IR stage, reference diffs, interface and artifact compatibility, buffer/event traces, unsupported cases, and a correlated latency breakdown of compile, load, allocation, transfer, queue, device, and synchronization time.

Build a ladder of models and measure the error between adjacent rungs

Benchmark hygiene

Fix the measurement boundary

State warm-up, repetitions, clocks, temperature, batch and arrival process, included transfers, compile/load time, and percentile calculation.

Counter hygiene

Reconcile rather than decorate

Check active + stall + idle against elapsed cycles; requested versus delivered bytes; issued versus retired work; overflow and multiplexing behavior.

Claim hygiene

Carry correctness with speed

A performance result is invalid if operator semantics, model accuracy, precision, fallback, preprocessing, or concurrency changed outside the declared experiment.

Product development and commercialization

Design the hardware and software interface for years of use, not one demo

A commercially useful NPU meets customer workloads within system cost and power while remaining integrable, secure, diagnosable, recoverable, upgradeable, and supportable across hardware and software revisions.

Value

End-to-end workload fit

Define customer models, quality, latency distributions, throughput, concurrency, cold start, memory, power, thermal, and fallback—not a single peak metric.

Economics

PPA and system cost

Trade compute, SRAM, NoC, DRAM bandwidth, die/package area, IP, yield, board memory, cooling, and engineering schedule against the actual workload mix.

Adoption

Compiler and SDK coverage

Make supported semantics discoverable, failures actionable, examples executable, results reproducible, and migration possible across target revisions.

Compatibility

Version every boundary

Negotiate capabilities and size-versioned messages; test old/new compiler, artifact, runtime, driver, firmware, and hardware combinations.

Trust

Isolation and safe failure

Operations

Provide health counters, trace/crash capture, firmware update and rollback, manufacturing tests, failure triage, deprecation policy, and support documentation.

SystemC/TLM and simulator development

Choose fidelity explicitly, then validate every timing claim

SystemC supplies an event-driven C++ simulation kernel. TLM-2.0 standardizes transaction interfaces. Neither makes a model cycle-accurate automatically: timing points, contention, arbitration, pipelines, and calibration are deliberate modeling choices.

UntimedFunctional modelSemantics, software enablement, golden state; highest speed
Register-accurateProgrammer viewMMIO, descriptors, interrupts, faults, firmware and driver flow

Approximately timedTLM nb_transportMultiple phases, overlap, backpressure, arbitration and contention

Blocking transport

Use LT to boot software early and sweep architecture parameters quickly

An initiator calls b_transport(payload, delay). The target executes the transaction in one blocking call and annotates coarse delay. A quantum keeper can let processes run ahead locally before synchronizing, improving simulation speed.

ModelRegisters, memories, commands, functional engines, interrupts, coarse DMA/compute latency, counters, errors.
Do not claimCycle-level arbitration, exact queue overlap, pipeline bubbles, or tail latency unless those mechanisms are explicitly represented and correlated.
Non-blocking transport

Use AT where request/response phases and contention change the answer

The base protocol uses nb_transport_fw/nb_transport_bw and four timing points: BEGIN_REQ, END_REQ, BEGIN_RESP, and END_RESP. Multiple transactions may be in flight, so payload lifetime, phase transitions, exclusion rules, and backpressure must be correct.

Add detailQueue depth, accept interval, service latency, arbitration, NoC/DRAM contention, DMA overlap, response ordering.
Pay the costMore scheduler interaction, protocol state, validation effort, slower simulation, and harder debugging.
Error is a metric

Correlate shared events, not unrelated summary numbers

Run the same artifact and input under the same clocks, memory parameters, firmware policy, and measurement boundary. Preserve command/engine IDs across TLM, RTL, FPGA, and silicon traces. Compare per-command cycles and bytes before comparing only total latency.

ReportAbsolute and relative error, bias by operator/shape, confidence, unmatched events, modeled and unmodeled causes.
RegressFreeze calibrated cases, retain raw traces/configuration, and alert when a change improves one class while degrading another.
Minimal LT target shapeSystemC / TLM-2.0 sketch
struct npu_target : sc_core::sc_module {
  tlm_utils::simple_target_socket<npu_target> socket{"socket"};

  SC_CTOR(npu_target) {
    socket.register_b_transport(this, &npu_target::b_transport);
  }

  void b_transport(tlm::tlm_generic_payload& tx,
                   sc_core::sc_time& delay) {
    const auto address = tx.get_address();
    auto* data = tx.get_data_ptr();

    if (!data || tx.get_data_length() != 4) {
      tx.set_response_status(tlm::TLM_BURST_ERROR_RESPONSE);
      return;
    }

    if (tx.is_write() && address == kDoorbell) {
      const auto command = read_u32(data);
      const auto arrival_delay =
          delay + sc_core::sc_time(20, sc_core::SC_NS);

      // A PEQ/event accepts work at the annotated arrival time;
      // it must not mutate visible state early.
      schedule_command(command, arrival_delay);
      delay = arrival_delay;
      tx.set_response_status(tlm::TLM_OK_RESPONSE);
      return;
    }

    tx.set_response_status(tlm::TLM_ADDRESS_ERROR_RESPONSE);
  }
};

Virtual-platform build order

  1. Implement command semantics and deterministic golden state without time.
  2. Add register map, memory targets, descriptors, interrupts, errors, and a driver-like test.
  3. Annotate compute and DMA latency; expose cycles, bytes, queue occupancy, and stalls.
  4. Add contention/backpressure only where sensitivity analysis shows it can change a decision.
  5. Replay seeded workloads against functional model and RTL; retain first-divergence traces.
  6. Calibrate with documented parameters, report error envelopes, and lock correlation regressions.

Also understand generic payload ownership, DMI and invalidation, debug transport, byte enables, streaming width, endianness, delta cycles, SC_THREAD/SC_METHOD behavior, and deterministic event ordering.
