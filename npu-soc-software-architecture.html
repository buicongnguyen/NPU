End-to-end architecture

The stack has two products: an executable artifact and a controlled device

Data-plane work carries tensors and commands. Control-plane work owns discovery, allocation, power, faults, version negotiation, and observability. A robust design makes both planes explicit.

The deployable artifact is the architectural hinge. It should identify the target and ABI, define constants and mutable buffers, carry command or kernel payloads, preserve graph-to-command debug maps, and include enough provenance to reproduce the compile. A binary blob without that contract is difficult to validate, debug, cache, or evolve.

Interface explorer

Every boundary needs ownership, invariants, observability, and versioning

Semantic boundary

Model to compiler

The importer must distinguish what the graph states, what it implies, and what remains dynamic. Operator names alone are not a contract: opset, attributes, layouts, quantization metadata, control flow, aliasing, and preprocessing assumptions all matter.

InputsModel IR, opset imports, shape constraints, constants, calibration profile, target policy.
Owned byExporter and importer jointly; neither side may silently reinterpret semantics.
InvariantsGraph validates, dtypes and ranks are legal, outputs match a framework reference within declared tolerance.
DiagnosticsUnsupported node path, source location, shape state, fallback reason, and minimal reproducer.
Deployable boundary

Compiler to runtime

The compiler freezes decisions that the runtime must execute without reconstructing compiler intent. Treat the artifact format as a versioned product, not an internal serialization accident.

PayloadTarget code, command templates, constants, relocations, buffer plan, entry points, debug map.
CompatibilityFormat version, runtime ABI, firmware ABI, target ID, feature bits, endianness, alignment.
SecurityBounds, signatures or hashes, section validation, integer-overflow checks, untrusted model policy.
ReproducibilityCompiler revision, pass configuration, model digest, target profile, deterministic build identifier.
Privilege boundary

Runtime to kernel driver

User space asks for capabilities, memory, queues, and synchronization. The kernel validates ownership and lifetime, maps memory for DMA, controls reset and power, and prevents one process from corrupting another.

ObjectsDevice, context, buffer, mapping, queue, fence/event, executable handle, telemetry stream.
SubmissionValidated descriptors, dependency fences, completion event, timeout, cancellation and reset policy.
MemoryCPU virtual, physical, and device-visible DMA addresses remain distinct; mapping lifetime is explicit.
ErrorsStable codes separate invalid input, out-of-memory, timeout, page fault, firmware fault, and device loss.
Asynchronous boundary

Driver to firmware

A ring, doorbell, and interrupt look simple until wraparound, reset, partial writes, stale generations, and malformed descriptors occur. Define producer and consumer indices, ownership transfer, memory ordering, and recovery together.

TransportShared command/completion rings, MMIO doorbells, generation counters, interrupt or polling mode.
OrderingDescriptor bytes become visible before the producer index; completion data becomes visible before signaling.
WatchdogPer-command deadline, heartbeat, fault snapshot, queue quarantine, warm reset, full reset escalation.
UpgradeFeature negotiation and backward-compatible message sizes protect driver/firmware combinations.
Hardware boundary

Firmware to NPU engines

The scheduler translates a command graph into engine work while preserving dependencies and resource limits. Register writes are only one part; descriptors, local-memory ownership, preemption points, and fault semantics complete the contract.

Work unitKernel or micro-op ID, tensor views, addresses, strides, tile counts, precision, dependency tokens.
ResourcesEngine slots, SRAM banks, DMA channels, NoC credits, scratch registers, power/clock state.
CompletionArchitectural write visibility, status code, performance counters, timestamp, fault address.
Model parityBit-accurate result, bounded cycle estimate, and named differences from RTL or silicon.
Observability boundary

Trace producer to debugger and profiler

Every layer, partition, command, DMA, interrupt, and counter sample needs a correlation path. Clock-domain conversion must carry source frequency, wrap width, synchronization samples, and uncertainty.

NPU runtime software

The runtime owns object lifetime and asynchronous truth

Session state

Create, load, bind, submit, retire, destroy

Each transition has preconditions. Destroy waits for or cancels in-flight work by policy; a failed device transitions the session to a terminal state instead of pretending later calls can succeed.

Queue semantics

Ordering must be named

Define in-order versus out-of-order execution, cross-queue dependencies, host visibility, event reuse, callback context, timeout behavior, and whether cancellation is best effort or guaranteed.

Memory objects

Ownership is separate from visibility

A buffer can be alive but not coherent, mapped but not device-visible, or device-visible but not writable. State transitions should name mapping, cache maintenance, import/export, and fence ownership.

Compiler architecture beyond basic rewrites

Scheduling and memory planning solve one coupled resource problem

Graph fusion, tiling, layouts, and quantization are already covered in NPU Practice. Here the focus is pass contracts, constrained scheduling, and offset assignment for a real target.

LegalizeExpose target formsNormalize ops without losing quantization intent
PartitionChoose ownershipPrice fallback boundaries and transfer cost

Critical-path list scheduling

Prioritize ready work by downstream urgency

Compute an earliest/latest schedule or remaining critical-path length. At each decision, choose a ready operation whose engine and buffers are available. Add transfer and bank-conflict cost to avoid a compute-only optimum.

Modulo or pipeline scheduling

Overlap repeated tiles and frames

Pick an initiation interval constrained by engine occupancy, DMA throughput, dependencies, and buffer reuse. Prologue, steady state, and epilogue schedules need separate validation.

Search and exact methods

Use costlier solvers on bounded regions

Dynamic programming, beam search, ILP, or constraint programming can optimize small fused regions. Production compilers usually combine exact local decisions with scalable heuristics and measured cost models.

Static arena planningLifetime and alignment aware
buffers = sort_by(size_descending, temporary_buffers)
active  = interval_index(buffers.live_range)
arena   = empty_free_list()

for buffer in buffers:
    legal = gaps_not_overlapping(active, buffer.live_range)
    legal = enforce_alignment_bank_and_address_class(legal, buffer)
    offset = best_fit_with_fragmentation_cost(legal, buffer.size)

    if offset is NONE:
        offset = grow_arena_or_spill(buffer)

    assign(buffer, offset)
    reserve_for_lifetime(arena, buffer.live_range, offset, buffer.size)

verify_no_illegal_aliases(arena)
emit_offsets_and_peak_usage(arena)

Allocator proof obligations

  • Buffers overlap in address only when lifetimes do not overlap or an in-place alias is explicitly legal.
  • Sub-byte and packed formats still satisfy engine and DMA alignment requirements.
  • Bank selection respects concurrent access patterns, not only total capacity.
  • Dynamic dimensions have bounded arenas, shape classes, or a runtime allocation path.
  • Spills include transfer commands, synchronization, and performance cost in the schedule.
  • The emitted peak is independently recomputed by a verifier before packaging.

Hardware C++ modeling and integration

One model cannot be fast, bit-exact, cycle-exact, and easy to change at once

Integration rule: do not let the performance model become a second undocumented ISA. Reuse the same generated register, descriptor, opcode, and trace definitions across compiler, runtime, firmware, model, and debugger whenever the build system permits it.

Hardware metric profiling

Power, cycles, latency, and bandwidth need a shared measurement frame

"Clock latency" is ambiguous. Report device cycles, clock frequency, queue delay, execution duration, and end-to-end wall time separately. Correlate each value to shape, batch, precision, power mode, and temperature.

Network layer step debugger

"Next layer" is a mapping problem, not a hardware primitive

Compilers fuse, split, reorder, tile, and overlap graph operations. A debugger should expose several step granularities and preserve the many-to-many map between source nodes, compiled regions, commands, and engines.

Logical step

Framework or ONNX node

Best for model authors. The debugger materializes a semantically meaningful output even when the implementation fused it away, which may require an instrumented recompile.

Compiled step

Partition or fused region

Best for compiler triage. It compares region inputs/outputs, target support, layouts, precision, and reference execution without exposing every tile.

Device step

Command, kernel, or micro-op

Best for runtime, firmware, and hardware work. It exposes queue dependencies, addresses, cycles, engine state, counters, and fault snapshots.

GitHub Actions distinguishes dependency caches from workflow artifacts: caches accelerate reproducible work, while artifacts retain outputs such as binaries, logs, traces, coverage, and reports. A failed NPU test should upload the replay package as an artifact, not hide it inside an ephemeral runner log.
